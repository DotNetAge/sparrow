简介​

本技术文档围绕仓库中的 Saga 协调器实现，系统阐述 Saga 模式在该代码库中的落地方式，包括：

  • 事务步骤的编排与执行
  • 补偿事务机制与回滚策略
  • 分布式事务协调思路与状态持久化
  • 长事务的分解、事务边界定义与状态管理
  • 并行执行、条件执行与异常处理
  • 适用场景、设计原则与性能优化建议

该实现采用整洁架构分层，协调器位于适配器层，服务层负责核心业务逻辑，仓储层负责状态持久化，任务调度器提供并发/顺序执行能力（与 Saga 协调器互补）。

项目结构​

围绕 Saga 协调器的关键模块如下：

  • 适配器层：对外暴露简化的协调器接口，封装服务层与仓储层
  • 用例层：实现 Saga 事务的核心业务逻辑（注册处理器、执行步骤、补偿）
  • 实体层：定义事务与步骤的数据结构及状态常量
  • 仓储层：提供通用仓储接口与内存仓储实现
  • 任务调度器：提供并发/顺序执行能力（可与 Saga 协调器配合）
  • 协调器（Coordinator）：位于适配器层，提供注册处理器/补偿器、执行事务、查询事务状态的简化接口；内部持有用例层的 SagaService。
  • SagaService：用例层核心，负责事务生命周期管理、步骤执行、补偿回滚、状态持久化。
  • Transaction/Step：实体层数据模型，描述事务与步骤的元数据与状态。
  • MemoryRepository：仓储层内存实现，提供事务实体的保存、更新、查询等操作。
  • TaskScheduler：任务调度器接口，提供并发/顺序执行能力，支持任务状态管理、重试、过期清理等。

下面的序列图展示了从协调器到服务层再到仓储层的调用链路，以及补偿流程。

协调器（Adapter Layer）​

  • 职责：对外提供简化的注册与执行接口；默认使用内存仓储；保持向后兼容的类型别名与状态常量。

  • 关键点：

    • 注册处理器与补偿器：通过服务层注册步骤名称与对应处理函数。
    • 执行事务：委托服务层执行，返回事务实体与错误。
    • 查询事务：委托服务层按 ID 查询事务状态。

  • 职责：核心业务逻辑，负责事务生命周期管理、步骤执行、补偿回滚与状态持久化。

  • 关键流程：

    • 创建事务实体并保存初始状态
    • 将状态置为运行中并保存
    • 顺序执行每个步骤：查找注册的处理器并执行；成功则更新进度并保存；失败则触发补偿
    • 补偿：从失败步骤反向执行补偿处理器（若存在），最终标记为补偿完成并保存
    • 成功：将状态置为完成并保存
    • 查询：按 ID 从仓储读取事务
  • Transaction：事务实体，包含 ID、名称、步骤列表、当前进度、状态、创建/更新时间。
  • Step：步骤实体，包含名称、处理器名称、载荷、补偿处理器名称。
  • 状态常量：pending、running、completed、failed、compensated。
  • BaseRepository：定义通用仓储接口与默认实现骨架（抛出“未实现”错误），供具体仓储实现覆盖。
  • MemoryRepository：基于内存的完整实现，支持保存、更新、查询、批量操作、分页、条件查询、存在性检查等；在保存/更新时自动维护时间戳字段。
  • 并发调度器：支持最大并发数控制、任务超时、重试、过期清理、取消与状态查询。
  • 顺序调度器：保证同一时刻仅有一个任务执行，支持定时与周期性任务、指数退避重试、过期清理。
  • Task 接口与 TaskInfo：统一任务抽象，包含 ID、类型、状态、调度时间、重试次数、错误信息、TTL 等。
  • 协调器依赖服务层；服务层依赖仓储接口与实体模型；仓储层提供内存实现。
  • 任务调度器独立于 Saga 协调器，但两者可组合使用：Saga 负责长事务编排与补偿，任务调度器负责任务级并发/顺序执行与重试。

  • 事务持久化：每次步骤成功都会保存进度，频繁写入可能成为瓶颈。建议：

    • 在高吞吐场景下引入异步批处理或后台刷新策略（需扩展仓储层）
    • 评估是否需要更持久化的存储（如数据库/缓存）替代内存仓储

  • 补偿开销：补偿按失败步骤反向执行，若补偿复杂或外部依赖较多，应尽量幂等与快速失败

  • 并发与顺序：对于长事务内的子任务，可结合任务调度器使用并发/顺序模式，避免阻塞主事务流程

  • 超时与重试：任务调度器内置超时与重试机制，合理设置超时与最大重试次数，避免雪崩

  • 事务状态异常

    • 检查仓储保存是否成功（保存/更新错误会被包装为仓储错误）
    • 确认事务状态流转：pending -> running -> completed/failed -> compensated

  • 步骤处理器缺失

    • 若注册的处理器名称与步骤中配置不一致，执行阶段会报“未注册处理器”的错误

  • 补偿未生效

    • 确认步骤配置了补偿处理器名称
    • 补偿执行被忽略错误（仅记录日志），请检查补偿处理器实现

  • 任务调度问题

    • 并发/顺序调度器支持取消、状态查询、过期清理与重试；检查任务 TTL、最大并发数、重试次数与下次重试时间

该实现以简洁的方式落地了 Saga 模式：通过步骤编排、处理器注册、状态持久化与补偿回滚，实现了长事务的可靠执行。配合任务调度器，可在更细粒度的任务层面实现并发/顺序执行与重试。生产环境中建议：

  • 替换内存仓储为持久化存储
  • 为补偿处理器提供幂等与可观测性
  • 明确事务边界与步骤粒度，避免过长事务
  • 结合任务调度器对子任务进行并发/顺序控制

设计原则与最佳实践​

  • 事务边界清晰：每个 Saga 事务聚焦单一业务目标，步骤数量与粒度适中
  • 幂等性优先：补偿处理器与步骤处理器应具备幂等能力
  • 异常即补偿：失败即触发补偿，补偿应尽可能快速、可恢复
  • 状态可见：通过仓储持久化状态，便于监控与重放
  • 可观测：为关键步骤与补偿提供日志与指标

适用场景​

  • 跨服务的长事务编排（如订单-库存-支付）
  • 需要补偿回滚的分布式操作
  • 业务流程可拆分为明确的步骤序列
