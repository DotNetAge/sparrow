简介​

本文件面向多数据库支持场景，系统化阐述 Sparrow 通用 SQL 数据库仓储的设计与实现，覆盖以下主题：

  • 抽象设计与接口契约：通过泛型仓储接口与基础仓储抽象，统一不同数据库实现的 CRUD 与查询能力。
  • 数据库无关性：通过统一的实体接口与查询选项模型，屏蔽底层差异。
  • SQL 生成策略：动态 SQL 构建、参数绑定、条件拼装、分页与排序。
  • 方言支持与连接管理：支持 sqlite3 与 PostgreSQL 的不同占位符与语法差异；连接池与事务管理。
  • 动态 SQL 与参数绑定：IN 子句、LIKE、NULL 条件等动态拼装与安全绑定。
  • 查询优化：软删除过滤、默认排序、LIMIT/OFFSET、COUNT 统计。
  • 错误处理与重试：统一的仓储错误类型、重试策略与优雅关闭。
  • 迁移与版本兼容：通过测试驱动的数据库容器与 DDL 变更验证。

项目结构​

围绕“仓储”这一核心域，项目采用分层与按领域职责划分的组织方式：

  • usecase：定义仓储接口与查询模型（泛型、条件、排序、分页）。
  • entity：定义实体接口与基础实体结构体。
  • persistence/repo：具体仓储实现（基于 database/sql 的通用实现与基于 sqlx 的 PostgreSQL 实现）。
  • bootstrap：应用启动、容器注册、数据库连接配置与重试机制。
  • config：数据库连接配置模型与 DSN 生成。
  • errs：统一错误类型（仓储错误）。
  • 仓储接口与查询模型
    • 仓储接口定义了 Save、Find、Update、Delete、批量操作、分页、统计、条件查询、随机取样等能力。
    • 查询模型包括 QueryOptions（条件、排序、分页）、QueryCondition（字段、运算符、值）。
  • 实体模型
    • Entity 接口要求具备 ID、创建/更新时间的读取与设置能力；BaseEntity 提供标准字段与构造函数。
  • 数据库连接与配置
    • SQLConfig 提供 Driver、Host、Port、User、Password、Dbname 等字段，并生成 DSN。
    • bootstrap.Database 将 SQLDB、Redis、Badger 统一注入到容器中。
  • 错误模型
    • RepositoryError 提供统一的仓储错误封装，包含实体类型、操作、ID、消息与原因。

通用 SQL 仓储通过“接口 + 泛型 + 基础抽象”的方式，为不同数据库提供一致的 API 表面。database/sql 实现适用于 SQLite 等轻量场景；sqlx 实现针对 PostgreSQL 提供更强的类型与扫描能力。

SqlDBRepository[T]：基于 database/sql 的通用实现​

  • 设计要点
    • 通过反射推断表名与实体类型，支持软删除检测（DeletedAt 字段）。
    • 统一的 Save：若存在则更新，否则插入；自动维护 CreatedAt/UpdatedAt。
    • 批量 SaveBatch：在单个事务内循环处理，保证一致性。
    • 查询族：支持按 ID、ID 列表、字段值、条件组合、分页、统计、随机取样。
    • 条件构建：EQ/NEQ/GT/GTE/LT/LTE/LIKE/IN/IS_NULL/IS_NOT_NULL。
  • SQL 生成与参数绑定
    • 占位符：?（SQLite/MySQL），IN 子句动态生成占位符，LIKE 自动包裹通配符。
    • 软删除：查询时自动追加 deleted_at IS NULL 条件。
    • 分页：LIMIT ? OFFSET ?，默认按 created_at DESC 排序。
  • 事务与连接
    • Save/SaveBatch/DeleteBatch 显式开启事务，异常时回滚，成功时提交。
    • 通过 *sql.DB 管理连接池与生命周期。
  • 错误处理
    • 使用统一的 RepositoryError 包装操作、实体类型、ID 与消息。
    • 对 sql.ErrNoRows 进行特判，返回“未找到”错误。
  • 设计要点
    • 使用 *sqlx.DB 与 QueryxContext/Select，支持更丰富的扫描与返回能力。
    • 支持 PostgreSQL 特有的占位符 $1/$2 与数组、JSONB 等类型处理。
    • 提供软删除（更新 deleted_at）与硬删除（Delete 方法）两种删除策略。
    • 批量操作在事务中执行，显式回滚与日志记录。
  • SQL 生成与参数绑定
    • 占位符：$1/$2，IN 子句动态生成 $n。
    • 字段名校验：isValidFieldName 防止注入风险。
    • 数组与 JSONB：通过自定义类型与扫描函数处理。
  • 事务与连接
    • 使用 *sqlx.Tx，支持 GetContext/QueryRowContext 等方法。
  • 错误处理
    • 统一 RepositoryError；对 RowsAffected 为 0 的删除场景返回“未找到”。
  • 条件拼装
    • buildCondition 将 QueryCondition 转换为 SQL 片段与参数切片，支持 IN、LIKE、NULL 等。
  • 分页与排序
    • FindWithConditions 自动拼接 WHERE/ORDER BY/LIMIT/OFFSET，默认按 created_at DESC。
  • 统计
    • CountWithConditions 与 CountByField/Count 统一走 WHERE 条件与 COUNT(*)。

  • 连接池

    • database/sql 通过 *sql.DB 管理连接池；bootstrap 中注册 SQLDB，便于全局共享。

  • 事务

    • Save/SaveBatch/DeleteBatch 在单事务内执行，异常回滚，成功提交。
    • PostgreSQL 实现使用 BeginTxx/BeginTx，增强上下文控制。

  • 优雅关闭

    • App.Start/Shutdown 与 CleanUp 管理子进程与资源释放；重试 goroutine 通过 cancel 控制退出。

  • 统一错误

    • RepositoryError 提供实体类型、操作、ID、消息与原因，便于定位与日志追踪。

  • 重试机制

    • App 内部启动重试 goroutine，对启动失败的子进程进行指数退避重试（最多 10 次，上限 1 分钟）。

  • 测试验证

    • SQLite/PostgreSQL 测试覆盖 Save、Find、Update、Delete、批量操作、分页、条件查询、随机取样等场景。

  • 组件耦合

    • 仓储实现依赖 usecase 接口与实体接口，低耦合高内聚。
    • database/sql 与 sqlx 作为外部依赖，分别服务于通用 SQL 与 PostgreSQL。

  • 外部依赖

    • database/sql、github.com/jmoiron/sqlx、测试容器 testcontainers 等。

  • 可能的循环依赖

    • 当前结构清晰，未发现循环导入；仓储实现仅依赖 usecase 与 entity。
  • SQL 生成与参数绑定
    • IN 子句按元素数量动态生成占位符，避免字符串拼接引发 SQL 注入。
    • LIKE 条件自动包裹通配符，避免全表扫描。
  • 分页与排序
    • 默认按 created_at DESC 排序，建议在高频查询字段上建立索引。
    • LIMIT/OFFSET 适合中小规模分页；大规模场景建议使用游标分页或基于索引的键集分页。
  • 软删除
    • 查询自动过滤 deleted_at IS NULL，减少无用数据扫描；建议为 deleted_at 建立索引。
  • 连接池
    • database/sql 通过 *sql.DB 管理连接池；建议结合业务峰值合理配置最大连接数与空闲连接数。
  • 批量操作
    • SaveBatch/DeleteBatch 在单事务内执行，减少往返与锁竞争；注意单事务大小与超时设置。

故障排查指南​

  • 常见错误类型
    • RepositoryError：包含实体类型、操作、ID、消息与原因，便于快速定位。
  • 排查步骤
    • 检查实体 ID 是否为空（Save/Update/Delete/Exists 均要求非空 ID）。
    • 检查数据库连接与 DSN（Driver/User/Host/Port/Dbname）。
    • 检查表结构与字段映射（DeletedAt、CreatedAt、UpdatedAt 等）。
    • 检查事务状态（异常时是否正确回滚）。
    • 观察日志与重试行为（App 的重试 goroutine 会记录退避与最大重试次数）。
  • 测试验证
    • 使用 sqldb_test 与 postgres_test 验证 Save/Find/Update/Delete/批量操作/分页/条件查询/随机取样等场景。

Sparrow 的通用 SQL 仓储通过“接口 + 泛型 + 基础抽象”，在保证数据库无关性的前提下，提供了完善的 CRUD、查询、分页、统计与条件构建能力。database/sql 实现适配 SQLite 等轻量场景，sqlx 实现强化了 PostgreSQL 的类型与扫描能力。配合统一的错误模型、事务管理与重试机制，能够在多数据库环境下提供一致且可靠的仓储体验。
