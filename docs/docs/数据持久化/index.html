简介​

本文件面向数据持久化开发者，系统性梳理 Sparrow 数据持久化体系：仓储模式实现、多数据源支持、事务管理机制，以及各存储后端（内存、Redis、PostgreSQL、MongoDB、BadgerDB）的特性与适用场景。文档同时给出仓储接口设计原则、数据访问模式、查询优化策略、迁移与备份恢复思路及性能调优建议。

项目结构​

Sparrow 的持久化层采用“泛型仓储 + 多后端适配”的架构：

  • usecase 层定义通用仓储接口与查询模型
  • entity 层定义实体契约
  • persistence/repo 下为各存储后端的具体实现
  • bootstrap 与 config 提供容器与配置装配能力
  • 仓储接口与查询模型
    • 仓储接口统一定义 CRUD、批量操作、分页、条件查询、随机取样等能力
    • 查询模型包括条件、排序、分页参数
  • 基础仓储
    • 提供默认实现骨架与错误处理模板，具体后端实现复用
  • 实体契约
    • 统一的实体接口，要求具备 ID、创建/更新时间字段的访问与设置能力

Sparrow 的持久化架构遵循“接口隔离 + 后端适配”的设计：

  • 通过泛型仓储接口屏蔽底层差异
  • 各后端独立实现，保证扩展性与可替换性
  • 事务管理在 SQL/PostgreSQL 场景由实现层显式控制
  • 配置与容器负责实例装配与生命周期管理

仓储接口设计原则​

  • 明确职责：单一接口承载完整数据访问能力，避免过度细分

  • 统一约定：所有实现遵循相同的上下文、错误与返回规范

  • 可扩展性：通过泛型与查询模型扩展新能力而不破坏现有契约

  • 一致性：分页、条件、排序在多后端保持一致的行为预期

  • 单实体：Save/FindByID/Update/Delete/Exists

  • 批量：SaveBatch/DeleteBatch/FindByIDs

  • 列表：FindAll/FindWithPagination/Count

  • 条件查询：FindByField/FindByFieldWithPagination/CountByField/FindWithConditions/CountWithConditions

  • 随机取样：Random

  • 索引与扫描

    • Redis/MongoDB/Badger：无原生索引，条件查询为全表扫描，建议配合分页与过滤范围
    • PostgreSQL：支持字段索引与复合索引，条件查询与排序可利用索引提升性能

  • 分页与排序

    • 统一使用 Limit/Offset 或游标方式，避免一次性加载全量数据
    • PostgreSQL 默认按创建时间倒序，可按业务字段建立索引

  • 批量操作

    • Redis 使用 Pipeline
    • PostgreSQL/SQL 采用事务包裹批量插入/更新

  • 随机取样

    • PostgreSQL 使用 RANDOM()
    • MongoDB 使用 $sample
    • Redis/Badger 通过随机键选择或全量扫描后随机抽取

  • PostgreSQL

    • SaveBatch/批量插入/更新在事务内执行，失败自动回滚
    • 软删除：通过标记 deleted_at 字段实现

  • SQL 通用仓储

    • Save/SaveBatch/批量操作均在事务中执行
    • 支持软删除（存在 DeletedAt 字段）或硬删除

  • Redis/Badger/MongoDB

    • 未内置事务抽象，需在上层业务用例中协调一致性
    • Redis 支持 Pipeline 提升批量吞吐与原子性

内存存储（内存缓存/临时数据）​

  • 特点

    • 完整 CRUD 与批量操作
    • 并发安全（读写锁）
    • 支持随机取样、分页、条件查询（全表扫描）

  • 适用场景

    • 测试环境、临时数据、缓存层（非持久）

  • 特点

    • 键空间前缀隔离实体类型
    • TTL 支持、Pipeline 批量
    • 全表扫描的条件查询

  • 适用场景

    • 缓存、会话、消息队列事件总线、事件存储（ESDB）

  • 特点

    • 完整 ACID 事务、软删除、索引与复杂查询
    • 支持批量操作与条件查询

  • 适用场景

    • 主数据、审计日志、强一致要求的数据

  • 特点

    • 集合命名规则、BSON 文档存储
    • 支持聚合与随机采样

  • 适用场景

    • 日志、事件溯源、半结构化数据

  • 特点

    • LSM-Tree 存储、键前缀扫描
    • 无软删除，全表扫描条件查询

  • 适用场景

    • 本地事件存储、边缘计算、嵌入式系统

  • 特点

    • 自动识别软删除字段，支持软/硬删除
    • 事务化批量操作

  • 适用场景

    • 兼容性需求、历史遗留数据库
  • 接口与实现
    • usecase/Repo 接口被所有具体仓储实现
    • BaseRepository 为各实现提供默认骨架
  • 实体契约
    • 所有仓储实现的泛型参数均实现 Entity 接口
  • 配置与容器
    • bootstrap/Database 统一封装 Redis、SQL、Badger 客户端
    • bootstrap/Container 提供依赖注入与单例缓存

  • Redis

    • 使用 Pipeline 批量写入，合理设置 TTL
    • 控制键前缀长度，避免过长前缀影响 Keys/MGET 性能

  • PostgreSQL

    • 为高频查询字段建立索引
    • 使用事务批量操作，减少往返开销
    • 分页查询避免深翻页（Offset 过大），考虑基于游标的分页

  • MongoDB

    • 使用聚合管道与索引，避免全表扫描
    • 合理使用 $sample 进行随机采样

  • BadgerDB

    • 通过键前缀扫描实现分页，注意全量遍历成本
    • 调整 Compactor 与阈值参数以平衡写放大与读放大

  • 通用

    • 批量操作优先使用事务或 Pipeline
    • 条件查询尽量缩小范围，结合分页与排序

  • 常见错误类型

    • RepositoryError：实体 ID 空、实体不存在、操作失败等
    • SQL/Redis/MongoDB 客户端错误：连接失败、超时、键不存在

  • 排查步骤

    • 核对实体 ID 与实体类型
    • 检查后端连接配置与网络连通性
    • 观察事务状态与回滚日志（PostgreSQL）
    • 分析批量操作的 Pipeline/事务边界

  • 建议

    • 为关键路径增加重试与熔断
    • 记录关键操作的上下文信息（traceId、实体ID）

Sparrow 的持久化体系通过泛型仓储接口与多后端适配，实现了统一的数据访问体验与灵活的部署选择。在实际工程中，应根据业务对一致性、可用性与性能的要求选择合适的后端，并结合索引、分页与批量操作策略提升整体表现。

数据迁移与备份恢复​

  • PostgreSQL

    • 使用逻辑备份工具进行定期快照
    • 迁移时注意软删除字段与索引重建

  • Redis

    • RDB/AOF 持久化策略与 AOF 重写
    • 迁移时注意键空间前缀与 TTL 一致性

  • MongoDB

    • 使用副本集与分片，定期备份
    • 迁移时注意集合命名与索引

  • BadgerDB

    • 文件系统级备份与恢复
    • 迁移时注意目录权限与阈值配置

  • 配置管理

    • 通过 config/* 统一管理连接参数
    • 环境隔离：开发/测试/生产使用不同配置

  • 监控与告警

    • 关键指标：连接数、QPS、慢查询、错误率
    • 后端特定：Redis 内存使用、PG WAL、MongoDB Oplog、Badger Compaction

  • 容器与装配

    • 使用 bootstrap/Container 管理依赖与生命周期
    • Database 统一封装多客户端，便于替换与测试
