简介​

本文件为 Sparrow 内存仓储实现的全面技术文档，聚焦于基于内存的通用仓储设计与实现。该实现采用泛型实体模型，提供完整的 CRUD 操作、批量操作优化、分页查询、条件查询、随机获取等功能，并通过读写锁保障并发安全。文档还涵盖反射操作的使用场景、错误处理策略、测试策略与示例，以及性能特征与优化建议，帮助开发者在测试与开发环境中高效使用内存仓储。

项目结构​

内存仓储位于持久化层的仓库模块中，遵循整洁架构的分层组织：

  • 仓储接口与基础能力：位于 usecase 层，定义通用仓储契约与查询选项。
  • 实体模型：位于 entity 层，提供统一的实体接口与基础实体结构体。
  • 内存实现：位于 persistence/repo 下，实现基于内存的仓储逻辑。
  • 错误处理：位于 errs 层，提供统一的仓储错误类型。
  • 工具与辅助：位于 utils 层，提供排序与随机等工具方法。
  • 应用集成：位于 bootstrap 层，提供应用容器与仓库解析能力。
  • 泛型内存仓储 MemoryRepository[T]
    • 基于泛型实体类型，实现完整的 CRUD、批量操作、分页、条件查询、随机获取等。
    • 使用读写锁保证并发安全，内部以 map[string]T 存储实体。
  • 基础仓储 BaseRepository[T]
    • 定义仓储契约与默认行为，提供查询选项、条件与排序结构。
  • 实体模型
    • Entity 接口与 BaseEntity 结构体，统一实体的标识与时间戳管理。
    • Task 实体作为示例实体，展示如何嵌入 BaseEntity 并实现 Entity 接口。
  • 错误处理
    • RepositoryError 提供统一的仓储错误类型，包含实体类型、操作、ID、消息与原因。
  • 工具与辅助
    • SortEntities 提供实体排序工具，尽量减少反射使用；Random 方法用于随机获取实体。

内存仓储采用“泛型 + 读写锁”的架构设计，结合反射与类型断言，在保证类型安全的同时提升性能与灵活性。整体交互如下：

内存仓储类图​

  • 写操作（Save、Update、Delete、SaveBatch、DeleteBatch、Clear）使用互斥锁（写锁），确保同一时刻只有一个写操作执行，避免竞态条件。

  • 读操作（FindByID、FindAll、FindByIDs、FindWithPagination、Count、FindByField、Exists、FindWithConditions、CountWithConditions、Random）使用读锁，允许多个读操作并发执行，提升读性能。

  • 在 Save 中先判断是否存在，再决定插入还是更新，内部仍使用互斥锁确保幂等性与一致性。

  • MemoryRepository[T] 与 BaseRepository[T] 均为泛型实现，要求 T 必须实现 Entity 接口。

  • 通过类型断言与反射结合，实现对实体字段（如时间戳）的自动设置与读取，保证类型安全与可扩展性。

  • NewMemoryRepository[T] 返回初始化的内存仓储实例，内部记录实体类型字符串，便于错误信息输出。

  • 插入与更新时，通过反射设置实体的 CreatedAt/UpdatedAt 字段，若实体未声明相应字段则忽略。

  • 条件查询与字段查询中，使用反射获取字段值并进行比较，支持多种基本类型与 time.Time 的比较。

  • Random 方法中，通过反射获取实体ID列表，再随机打乱后选取指定数量的实体。

  • Save：校验实体ID，若存在则更新，否则插入；插入时设置时间戳。

  • Update：校验ID并存在性，设置更新时间后更新。

  • FindByID：根据ID查找实体，不存在返回仓储错误。

  • FindAll：返回所有实体副本。

  • Delete：根据ID删除实体。

  • Exists：检查实体是否存在。

  • Count：返回实体数量。

  • SaveBatch：批量保存实体，一次性获取写锁，逐个设置时间戳并写入，避免多次加锁开销。

  • FindByIDs：批量查找，仅一次读锁，遍历ID列表并过滤不存在项。

  • DeleteBatch：批量删除，一次性写锁，逐个删除。

  • 以上操作均对空输入进行快速返回，减少无意义处理。

  • 分页查询 FindWithPagination：准备所有实体切片，处理 limit/offset 边界，返回指定范围。

  • 条件查询 FindWithConditions：遍历实体，逐个条件匹配，支持 EQ、NE、GT、LT、GTE、LTE 等操作符。

  • 字段查询 FindByField：按字段名与值进行匹配，支持类型转换与指针解引用。

  • 随机获取 Random：随机打乱实体ID列表，按需选取指定数量实体。

  • 统计 CountWithConditions：与条件查询类似，但只统计数量。

  • RepositoryError：统一的仓储错误类型，包含实体类型、操作、ID、消息与原因，便于定位问题。

  • 对于空ID、未找到实体、重复插入等情况，返回明确的错误类型与消息。

  • 基础仓储提供默认未实现错误，确保上层实现覆盖关键方法。

  • 单元测试覆盖 CRUD、批量操作、分页、条件查询、随机获取、并发插入等场景。

  • 并发测试验证读写锁在高并发下的正确性与性能表现。

  • 使用 Task 实体作为示例，确保测试与实际业务一致。

内存仓储的依赖关系清晰，职责分离明确：

  • MemoryRepository[T] 依赖 Entity 接口与 BaseEntity 结构体，确保实体具备统一标识与时间戳。
  • 基础仓储 BaseRepository[T] 提供查询选项与默认未实现方法，降低具体实现负担。
  • 反射与类型断言在内存仓储中被谨慎使用，优先通过类型断言减少反射成本。
  • 工具层 SortEntities 与 Random 方法为查询与排序提供辅助能力。

  • 并发模型

    • 读多写少场景下，读写锁显著提升吞吐量；写操作使用互斥锁，避免竞争。

  • 反射与类型断言

    • 插入/更新时通过反射设置时间戳，字段存在性检查与类型判断保证安全性；条件查询与字段查询中使用反射，但仅在必要时进行。

  • 内存使用

    • 以 map[string]T 存储实体，适合中小规模数据；大规模数据建议考虑外部存储或缓存层。

  • 分页与条件查询

    • FindWithPagination 与 FindWithConditions 会对全量数据进行遍历，适合小到中等规模数据；对于大规模数据，建议引入索引或外部存储。

  • 随机获取

    • Random 方法对实体ID进行随机打乱，时间复杂度 O(n)，适合实体数量适中的场景。

  • 常见错误类型

    • RepositoryError：包含实体类型、操作、ID、消息与原因，便于快速定位问题。

  • 常见问题与解决

    • 空ID导致的错误：确保实体ID非空后再调用 Save/Update/Delete/Exists。
    • 未找到实体：检查实体是否已保存，或确认ID是否正确。
    • 并发冲突：确保在高并发场景下使用读写锁，避免外部竞态。

  • 调试建议

    • 使用单元测试覆盖关键路径，关注并发测试与边界条件。
    • 对于条件查询与字段查询，检查字段名与类型是否匹配。

内存仓储通过泛型与读写锁实现了高性能、易扩展的通用仓储能力，适用于测试环境与不需要持久化的场景。其设计兼顾了类型安全、反射与类型断言的平衡，提供了完善的 CRUD、批量操作与查询功能。在实际使用中，应根据数据规模与并发特性选择合适的部署策略，并通过充分的单元测试与并发测试保障稳定性。

附录​

  • 使用指南
    • 创建内存仓储实例：调用 NewMemoryRepository[T]。
    • 基于实体接口：确保实体实现 Entity 接口，以便自动设置时间戳。
    • 查询与分页：利用 QueryOptions 与分页方法进行条件查询与分页。
    • 并发访问：在高并发场景下，读写锁能有效保障一致性与性能。
  • 性能优化建议
    • 控制实体数量：内存仓储适合中小规模数据，避免过大映射影响性能。
    • 减少反射使用：在实体设计中尽量暴露可直接访问的时间戳字段，减少反射开销。
    • 批量操作：优先使用 SaveBatch、FindByIDs、DeleteBatch 等批量方法，减少锁竞争。
    • 条件查询：对大规模数据，建议引入外部存储或索引以提升查询效率。
