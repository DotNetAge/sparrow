项目结构​

日志模块位于 pkg/logger，围绕 zap 与 lumberjack 实现高性能、可轮转的日志能力；配置由 pkg/config 提供，bootstrap 在应用启动阶段装配全局 Logger。

  • pkg/config/config.go

  • pkg/config/logger.go

  • pkg/config/loader.go

  • pkg/bootstrap/app.go

  • pkg/bootstrap/opts.go

  • Logger 包装器：对 zap.Logger 进行轻量封装，提供 Info/Error/Fatal/Panic/Debug/Warn 及其格式化变体，统一字段风格。

  • 开发模式日志器：控制台输出，彩色级别，短调用者路径，便于本地调试。

  • 生产模式日志器：基于 lumberjack 的文件轮转，JSON 编码，多目标输出（文件+控制台），支持调用者与堆栈追踪。

  • 配置体系：LogConfig 定义 mode/level/format/output/filename；全局 Config 提供默认值与环境变量覆盖；loader 支持多格式配置文件与 .env 加载。

  • 应用装配：App 在启动时加载配置并创建 Logger，贯穿整个生命周期；各子系统通过注入的 Logger 输出状态、错误与告警。

日志系统采用“配置驱动 + 模式切换 + 多编码器”的架构，确保开发与生产环境的一致体验与高可靠输出。

组件详解​

Logger 包装器与级别管理​

  • 提供 Info/Error/Fatal/Panic/Debug/Warn 及其格式化方法，统一字段键名（如 timestamp、level、caller、message、stacktrace）。
  • 级别映射：开发模式按 LogConfig.Level 映射到 Debug/Info/Warn/Error/Fatal；生产模式按相同规则设置最终级别。
  • 运行模式：当 mode=prod|production 时启用生产模式日志器；否则使用开发模式日志器。

  • lumberjack 配置：文件大小阈值、备份数量、最大保存天数、是否压缩、使用本地时间命名。

  • 编码器：JSON 格式，字段键名标准化；时间、级别、调用者、耗时等均采用统一编码器。

  • 输出目标：文件与控制台双写，便于生产与审计。

  • 调用者与堆栈：启用 caller 与错误级别堆栈输出，便于问题定位。

  • 日志级别：依据 LogConfig.Level 动态映射。

  • 适用场景：本地开发、快速调试、CI 环境。

  • loader：支持多格式配置文件、.env 加载、环境变量覆盖、Snake 命名空间替换。

  • 编码与输出

    • 生产模式使用 JSON 编码，便于下游日志收集与检索；控制台仅在开发模式启用，避免生产环境冗余输出。
    • lumberjack 轮转参数建议结合磁盘容量与 IO 能力调整：单文件大小、备份数量、最大保存天数、压缩开关。

  • 日志级别

    • 严格区分 Debug/Info/Warn/Error/Fatal，避免在生产环境输出过多 Debug 信息。
    • 对高频路径使用 Info/Warn 记录关键状态，Error 记录异常，Fatal 用于不可恢复错误。

  • 调用者与堆栈

    • caller 与 stacktrace 有助于定位问题，但会带来额外开销；仅在错误级别启用堆栈，减少常规日志的性能影响。

  • 多目标输出

    • 文件与控制台双写在生产环境可能增加 IO 压力；建议仅在必要时开启控制台输出，或通过外部日志代理集中采集。

故障排查指南​

  • 启动失败
    • 若配置加载失败或校验失败，App.NewApp 会抛出错误；检查配置文件格式、键名与默认值。
    • 关注日志器创建失败的情况，确认 LogConfig.mode/level/filename 是否合理。
  • 运行期错误
    • 各组件初始化失败时会通过 Logger 输出错误字段，例如数据库连接失败、NATS 连接失败等；结合错误字段与堆栈定位问题。
  • 服务器关闭与清理
    • App 在优雅关闭时会记录关闭状态与清理结果；若清理失败，查看错误字段并确认资源释放顺序。

Sparrow 的日志系统以配置驱动为核心，通过开发/生产两种模式适配不同环境需求；借助 zap 与 lumberjack 实现高性能、可维护的日志输出。配合统一的结构化字段与多目标输出，既满足本地开发的便捷性，也满足生产环境的审计与分析需求。建议在生产环境中启用文件轮转与 JSON 编码，合理设置日志级别与采样策略，并结合外部监控平台实现告警与可视化。

附录​

日志级别与运行模式对照​

  • 开发模式：按 LogConfig.Level 映射 Debug/Info/Warn/Error/Fatal。

  • 生产模式：按相同规则设置最终级别，启用文件轮转与 JSON 编码。

  • 标准字段键名：timestamp、level、caller、message、stacktrace。

  • 编码器统一：生产模式使用 JSON 编码器，开发模式使用控制台编码器。

  • 开发环境

    • mode=dev，level=debug/info，format=console，output=stdout，filename 可选。

  • 生产环境

    • mode=prod|production，level=info/warn/error，format=json，output=file，filename 指向持久化目录。
    • 合理设置轮转参数：单文件大小、备份数量、最大保存天数、压缩开关。

  • 环境变量

    • 使用 .env 与 AutomaticEnv，确保敏感信息与运行时参数可控。

  • 日志采集

    • 将生产模式日志输出到文件，使用日志代理（如 Fluent Bit/Filebeat）采集并转发至 ELK/OTLP 等平台。

  • 告警

    • 在日志平台中基于 level=message/caller 等字段建立告警规则，结合业务指标与链路追踪进行联动。

  • 可视化

    • 利用日志平台的仪表板对错误趋势、响应时间分布、调用者热点进行可视化分析。

安全与隐私、合规​

  • 敏感信息脱敏
    • 避免在 message/caller 中直接记录明文密码、Token、身份证等敏感数据；必要时进行脱敏或哈希处理。
  • 日志访问控制
    • 限制日志文件读取权限，仅允许运维与审计人员访问；生产日志应加密存储与传输。
  • 合规要求
    • 满足 GDPR、网络安全等级保护等要求，明确日志保留期限与销毁流程；定期进行合规审计。
