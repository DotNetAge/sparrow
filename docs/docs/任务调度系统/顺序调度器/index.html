简介​

本文件系统性阐述顺序调度器的设计与实现，重点覆盖以下方面：

  • 任务队列管理与执行顺序保证
  • 串行化处理与并发对比
  • 任务排队机制、执行顺序控制、任务依赖处理
  • 与并发调度器的区别及适用场景
  • 配置方法、性能考量、错误处理策略
  • 实际使用场景与代码示例路径

顺序调度器适用于需要严格串行执行的任务场景，例如定时任务、周期性任务、需要避免资源竞争或状态冲突的批处理任务等。

项目结构​

顺序调度器位于任务调度模块中，与并发调度器、混合调度器共同构成统一的任务执行基础设施。其核心文件如下：

  • 顺序调度器实现：pkg/tasks/sequential_scheduler.go
  • 并发调度器实现：pkg/tasks/concurrent_scheduler.go
  • 调度器接口与数据模型：pkg/tasks/core.go
  • 任务构建器：pkg/tasks/builder.go
  • 调度器包装器（统一入口）：pkg/tasks/wrapper.go
  • 混合调度器（按任务类型选择执行模式）：pkg/tasks/hybrid_scheduler.go
  • 单元测试与应用层测试：pkg/tasks/tasks_test.go、pkg/bootstrap/app_scheduler_test.go
  • 顺序调度器（sequentialScheduler）：单线程串行执行，保证任务间严格顺序；内置待执行队列、任务完成通知通道、任务TTL与最大任务数限制。
  • 并发调度器（concurrentScheduler）：多任务并发执行，受最大并发数限制；支持定时/重试任务的轮询处理。
  • 混合调度器（HybridScheduler）：根据任务类型或调度时间自动选择顺序或并发执行；支持配置任务类型到执行模式的映射。
  • 调度器包装器（SchedulerWrapper）：对外统一入口，负责任务提交、状态查询、取消、关闭等；支持默认重试次数、并发数配置、任务类型模式设置。
  • 任务构建器（TaskBuilder/builtTask）：封装任务ID、类型、调度时间、处理函数、回调、超时、重试等属性，生成可调度任务。

顺序调度器采用“执行器+监视器”的双协程架构：

  • 执行器（runTaskExecutor）：遍历待执行队列，按序取出任务并执行；执行期间标记isRunning，防止并发执行。
  • 监视器（runTaskMonitor）：每秒扫描定时任务与过期任务，将到期任务加入待执行队列；定期清理过期已完成任务。

顺序调度器（sequentialScheduler）​

  • 数据结构与关键字段
    • 任务映射与互斥锁：tasks、taskMutex
    • 待执行队列与互斥锁：pendingTasks、pendingMutex
    • 执行控制：isRunning、taskDoneChan、taskQueue、taskQueueClosed
    • 生命周期：startOnce、stopChan、wg
    • 任务生命周期：taskTTL、maxTaskCount
  • 核心流程
    • Schedule：校验任务、存储任务、入待执行队列、通知执行器、尝试入任务队列
    • Cancel：调用任务取消回调、更新状态、从待执行队列移除
    • runTaskExecutor：按序取出任务、执行、更新状态、处理重试/周期性
    • runTaskMonitor：定时扫描、到期任务入队、清理过期任务
    • executeTask：超时控制、错误处理、重试指数退避、周期性任务重新调度
    • 清理策略：cleanupExpiredTasks（按TTL）、cleanupOldTasks（按最旧已完成）

  • 设计要点：最大并发数控制、活跃任务计数与条件变量、定时/重试任务轮询处理、过期任务清理。

  • 关键流程：Schedule（即时任务直接执行）、Cancel（更新状态并触发回调）、executeTask（并发槽位等待与释放）。

  • 设计要点：根据任务类型或调度时间选择顺序/并发；合并两个调度器的状态与列表；仅对并发调度器开放最大并发数设置。

  • 关键流程：Schedule（类型映射/定时/周期性默认顺序）、Cancel（双调度器尝试）、GetTaskStatus（并发优先）、ListTasks（合并列表）。

  • 设计要点：统一对外入口；支持默认重试次数、最大并发数、任务类型执行模式映射；提供便捷的定时/周期性任务提交方法。

  • 关键流程：RunTask系列（立即/定时/周期性）、RunTypedTask（按类型提交）、SetTaskTypeMode（动态切换模式）、SetMaxConcurrentTasks（并发调度器生效）。

  • 设计要点：链式配置任务ID、类型、处理函数、回调、调度时间、超时、重试、优先级；生成内置任务实现。

  • 关键流程：Build（校验参数/计算执行时间）、builtTask（实现Task与TaskInfoProvider接口）。

  • 接口契约

    • TaskScheduler：统一的调度器接口，顺序/并发/混合调度器均实现该接口。
    • Task/TaskInfoProvider：任务与任务信息接口，builtTask实现两者。

  • 组件耦合

    • 顺序调度器内部依赖任务信息接口以读取/更新状态；依赖任务超时与重试配置。
    • 包装器依赖调度器接口，向上提供统一API；向下委托给具体调度器实现。
    • 混合调度器组合顺序/并发调度器，按规则分发任务。

  • 顺序执行的吞吐量：单线程串行，吞吐量受限于单任务处理速度；适合对顺序敏感或资源竞争敏感的场景。

  • 内存与清理：维护任务映射与待执行队列，支持TTL与最大任务数限制；过期任务与最旧已完成任务清理降低内存压力。

  • 通道与互斥：使用带缓冲的任务队列与通知通道，配合互斥锁保护共享状态，避免竞态。

  • 重试与超时：顺序调度器支持指数退避重试与任务超时控制，避免阻塞其他任务。

  • 对比并发调度器：并发调度器通过最大并发数与条件变量控制并行度，适合高吞吐、无状态或可并行的任务。

  • 任务未执行

    • 检查任务状态是否为等待且已到期；确认runTaskMonitor是否扫描到定时任务并加入待执行队列。
    • 确认执行器是否因isRunning为true而阻塞；检查taskDoneChan是否被正确发送/接收。

  • 任务卡住或超时

    • 检查任务Handler是否阻塞；顺序调度器会为任务创建带超时的上下文。
    • 观察重试计数与NextRetryAt；确认指数退避是否导致延迟。

  • 内存增长

    • 检查cleanupExpiredTasks与cleanupOldTasks是否按预期清理；调整taskTTL与maxTaskCount。

  • 并发/顺序混淆

    • 混合调度器会根据任务类型或调度时间选择执行模式；确认类型映射与调度时间配置。

顺序调度器通过严格的串行化执行与完善的任务生命周期管理，为需要顺序一致性与资源安全性的任务提供了可靠保障。结合混合调度器与调度器包装器，可在同一系统内灵活地为不同任务类型选择最优执行模式，并提供统一的配置与使用体验。

顺序调度器与并发调度器的区别与适用场景​

  • 顺序调度器

    • 优点：严格顺序、资源安全、易于调试与审计
    • 缺点：吞吐量受限、延迟较高
    • 适用：定时任务、周期性任务、状态机驱动、外部系统交互

  • 并发调度器

    • 优点：高吞吐、低延迟
    • 缺点：需注意竞态与资源冲突
    • 适用：无状态或可并行的任务、批量数据处理

性能与错误处理策略​

  • 性能

    • 合理设置任务TTL与最大任务数，避免内存膨胀
    • 顺序执行适合小批量、高价值任务；大批量任务建议并发或混合模式

  • 错误处理

    • 顺序调度器支持指数退避重试与任务超时；失败后可进入重试或失败状态
    • 监视器定期清理过期任务，避免僵尸任务堆积
