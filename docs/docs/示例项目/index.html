简介​

本示例项目基于 Sparrow 框架，演示了如何使用 DDD（领域驱动设计）、CQRS（命令查询职责分离）与事件溯源模式构建一个可扩展的订单系统。项目采用整洁架构分层：领域层（实体、值对象、事件）、用例层（命令、查询、仓储接口、应用服务）、适配器层（HTTP 处理器、路由）与基础设施层（内存仓储、投影等）。示例同时提供了简化版 demo 与完整版实现，便于快速上手与深入学习。

项目结构​

示例项目位于 examples/order-system，核心目录与文件如下：

  • cmd：应用入口，包含 demo 版本与完整版本
  • configs：配置文件（应用、日志、数据库、事件总线、HTTP）
  • pkg：
    • domain/entities：领域模型（聚合根、值对象、事件、错误）
    • usecases：用例层（命令、查询、仓储接口、应用服务）
    • adapters：适配器层（HTTP 处理器、路由）
    • infrastructures：基础设施层（内存仓储等）

核心组件​

  • 领域模型
    • 聚合根：Order（订单），包含客户ID、订单项、总价、状态、配送地址
    • 值对象：Address（地址），包含街道、城市、州/省、邮编、国家
    • 领域事件：OrderCreated、OrderConfirmed、OrderCancelled、OrderShipped、OrderDelivered
    • 错误类型：OrderStatusError（订单状态错误）
  • 用例层
    • 命令：CreateOrderCommand、ConfirmOrderCommand、CancelOrderCommand、ShipOrderCommand、DeliverOrderCommand
    • 仓储接口：OrderRepository（支持按客户ID、状态查询）
    • 应用服务：OrderApplicationService（封装业务流程，协调事件存储与发布）
  • 适配器层
    • HTTP 处理器：OrderHandler（接收请求，构造命令/查询，调用应用服务）
    • 路由：SetupRouter（定义 /api/v1/orders 相关路由）
  • 基础设施层
    • 内存仓储：InMemoryOrderRepository（基于内存的简单实现）

架构总览​

该示例严格遵循整洁架构，关注“依赖倒置”与“用例为中心”的设计思想：

  • 领域层不依赖外部框架，仅包含业务规则与不变量
  • 用例层封装业务流程，协调事件存储与发布
  • 适配器层负责输入输出（HTTP），解耦具体框架
  • 基础设施层提供仓储与事件总线等实现

组件详细分析​

领域模型设计​

  • 聚合根 Order
    • 负责订单生命周期的状态转换与业务规则校验
    • 通过 AddEvent 记录领域事件，支持事件溯源
  • 值对象 Address
    • 作为不可变数据载体，提供基本校验
  • 领域事件
    • 每次状态变更生成对应事件，用于读模型重建与跨服务通信
  • 错误类型
    • 领域特定错误，便于在应用服务层统一处理

用例与应用服务​

  • 命令层
    • 定义 CreateOrderCommand、ConfirmOrderCommand、CancelOrderCommand、ShipOrderCommand、DeliverOrderCommand
  • 应用服务
    • OrderApplicationService 负责：
      • 参数校验（如地址有效性）
      • 聚合根创建与状态转换
      • 事件存储与事件发布
      • 标记事件为已提交
  • 仓储接口
    • OrderRepository 提供通用仓储能力与按客户ID/状态查询

适配器层（HTTP 接口）​

  • 路由
    • /health 健康检查
    • /api/v1/orders：创建、确认、取消、发货、收货、获取、列表
  • HTTP 处理器
    • 将请求参数绑定为命令/查询对象
    • 调用应用服务执行业务逻辑
    • 统一返回 JSON 响应

基础设施层（内存仓储）​

  • InMemoryOrderRepository
    • 基于通用内存仓储实现
    • 提供按客户ID与状态的过滤查询
    • 使用读写锁保证并发安全

状态流转与业务规则​

  • 订单状态机
    • Pending → Confirmed → Shipped → Delivered
    • 任意状态下均可 Cancelled（除已 Delivered 或已 Cancelled）
  • 业务规则
    • 确认前必须为 Pending
    • 发货前必须为 Confirmed
    • 收货前必须为 Shipped
    • 已完成状态不可再次变更

依赖关系分析​

  • 模块依赖
    • 示例模块 replace 指向本地 src，依赖 Sparrow 框架
  • 运行时依赖
    • Web 框架：Gin
    • 配置加载：YAML
    • 事件存储/发布：内存事件存储与事件总线
    • 仓储：内存仓储

性能考量​

  • 内存存储适合演示与小规模测试；生产环境建议替换为持久化仓储（如 PostgreSQL、MongoDB 等）
  • 事件存储与事件发布应考虑异步与幂等，避免阻塞请求路径
  • 查询端可通过投影与索引优化读模型性能
  • 并发访问时注意仓储实现的锁粒度与一致性

故障排查指南​

  • 常见错误类型
    • 订单状态错误：当状态不符合业务规则时抛出领域错误
  • 排查步骤
    • 检查请求参数绑定与命令/查询对象构造
    • 确认事件存储保存与事件发布是否成功
    • 校验仓储实现的查询过滤逻辑
  • 建议
    • 在应用服务层集中处理错误并返回语义化错误信息
    • 对关键路径增加日志与指标监控

结论​

该示例以最小可用的方式展示了 DDD、CQRS 与事件溯源在订单系统中的落地实践。通过清晰的分层与职责划分，示例既易于理解又具备良好的扩展性。建议在掌握示例后，逐步替换为持久化仓储与事件总线实现，并结合业务场景完善投影与查询端能力。

API 接口说明​

  • 健康检查
    • GET /health
  • 订单管理
    • POST /api/v1/orders
    • POST /api/v1/orders/{id}/confirm
    • POST /api/v1/orders/{id}/cancel
    • POST /api/v1/orders/{id}/ship
    • POST /api/v1/orders/{id}/deliver
    • GET /api/v1/orders/{id}
    • GET /api/v1/orders

使用与扩展指引​

  • 快速开始
    • 进入示例目录，安装依赖并运行 demo 或完整版本
  • 切换持久化存储
    • 实现 OrderRepository 的持久化版本并在依赖注入容器中注册
  • 新增领域事件
    • 在领域层定义事件，聚合根中触发，投影中处理
  • 新增查询
    • 在用例层定义查询对象与视图模型，在应用服务中实现并在 HTTP 层暴露
