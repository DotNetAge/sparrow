简介​

本指南围绕 Sparrow 框架的命令模式实现，系统阐述命令在领域驱动设计（DDD）中的角色、命令与聚合根的交互机制、命令验证与业务规则检查、事件生成与发布、序列化与传输、幂等性与错误处理、事务管理，以及在 CQRS 架构中命令与查询的分离实践。文档以仓库中的实体、用例、事件存储、投影与消息发布模块为依据，结合类图、时序图与流程图，帮助读者快速理解并正确使用命令模式。

项目结构​

Sparrow 框架围绕“实体层-用例层-基础设施层”的整洁架构组织命令模式相关代码：

  • 实体层：定义命令接口、聚合根接口与基础聚合根、领域事件接口等
  • 用例层：定义执行器接口、事件存储接口、仓储接口等
  • 投影层：定义事件读取器、聚合索引器与投影器接口，支撑查询侧
  • 消息层：提供事件序列化与发布能力，连接事件存储与事件总线
  • 命令接口与基础命令
    • 命令接口提供命令标识与目标聚合标识的访问能力
    • 基础命令结构体提供标准字段，统一命令的 command_id 与聚合标识 id
  • 聚合根接口与基础聚合根
    • 聚合根接口定义聚合类型、聚合标识、版本控制、命令处理、事件应用与加载等能力
    • 基础聚合根提供版本号、未提交事件列表、事件添加与标记提交等通用行为
  • 领域事件接口
    • 定义事件唯一标识、类型、创建时间、聚合标识、聚合类型、版本等
  • 事件存储接口
    • 提供事件持久化、读取、聚合加载、快照与批量操作等能力
  • 执行器接口
    • 统一命令执行入口，将 action（命令）交由业务意图处理
  • 投影接口
    • 事件读取器、聚合索引器与投影器支撑查询侧视图构建
  • 事件发布器
    • 将领域事件序列化为通用事件并通过事件总线发布

命令模式在 DDD/CQRS 中的角色：

  • 命令侧（写模型）：命令经执行器进入用例层，由聚合根处理并产生领域事件；事件写入事件存储，随后通过事件发布器广播
  • 查询侧（读模型）：事件读取器从事件存储读取事件流，聚合索引器提供聚合根 ID 列表，投影器将事件流重放为视图供查询使用

命令定义与命名约定​

  • 命令接口与基础命令

    • 命令接口提供命令标识与聚合标识访问
    • 基础命令结构体统一字段：command_id 与 id（映射到聚合标识）
    • 命令处理注意事项：不得覆盖 GetAggregateID() 方法，避免破坏系统对 JSON/HTTP Body 的默认映射

  • 命名约定

    • 命令类型应体现业务意图（如“创建订单”、“确认订单”），便于事件溯源与审计
    • 命令字段命名遵循 JSON/表单标签约定，确保序列化与传输一致性

  • 命令处理流程

    • 执行器接收命令，调用聚合根的 HandleCommand，返回领域事件列表
    • 聚合根内部执行业务规则校验与状态变更，通过 AddEvent 将事件加入未提交事件列表
    • 事件存储保存未提交事件，聚合根标记事件已提交

  • 事件生成与应用

    • 领域事件接口定义事件元数据（ID、类型、聚合标识、版本等）
    • 基础聚合根在 AddEvent 中进行事件完整性校验与版本递增，保证事件溯源一致性

  • 验证与规则

    • 聚合根接口预留 Validate()，可在聚合状态层面进行业务规则校验
    • 基础聚合根在 AddEvent 中对事件 ID、类型、聚合标识一致性进行严格校验，防止事件流污染

  • 事件生成

    • 聚合根在 HandleCommand 中根据业务意图生成领域事件，事件携带聚合标识与版本
    • 事件版本与聚合根版本同步递增，确保事件溯源顺序与一致性

  • 序列化

    • 事件发布器将领域事件序列化为字节，再进行 Base64 编码，封装为通用事件结构
    • 通用事件包含事件 ID、事件类型（含聚合类型）、时间戳与负载（Base64 编码）

  • 传输

    • 通过事件总线发布通用事件，订阅方解码并应用事件

  • 执行

    • 执行器统一接收命令，调用聚合根处理并生成事件，随后持久化与发布

  • 幂等性

    • 建议在命令层引入命令 ID（command_id）与聚合标识，结合事件存储的期望版本控制，避免重复处理
    • 事件 ID 唯一性校验（基础聚合根 AddEvent 中对事件 ID 重复进行检查），有助于防止重复事件污染

  • 错误处理

    • 事件发布器对序列化失败进行错误包装，便于定位问题
    • 事件总线各实现对处理器返回错误采取非阻断策略，确保发布流程稳定

  • 事务管理

    • 事件存储接口提供基于期望版本的保存能力，结合聚合根版本控制，实现乐观并发控制
    • 建议在命令处理与事件保存之间使用本地事务，失败回滚以保证一致性

  • 命令侧（写模型）

    • 命令经执行器进入聚合根，生成领域事件并持久化

  • 查询侧（读模型）

    • 事件读取器从事件存储读取事件流
    • 聚合索引器提供聚合根 ID 列表
    • 投影器将事件流重放为视图，供查询使用

  • 全量投影

    • 全量投影器按事件类型与聚合类型遍历聚合根 ID，逐个投影生成视图集合

  • 命令设计

    • 命令类型明确、字段精简，避免过度耦合
    • 命令不可重写聚合标识获取方法，确保系统默认映射一致

  • 聚合根设计

    • 业务规则集中在聚合根内部，通过事件应用状态
    • 事件 ID 与类型必须唯一且与聚合标识一致

  • 事件存储

    • 使用期望版本控制实现乐观锁，减少并发冲突
    • 支持快照以加速大型聚合的重放

  • 查询优化

    • 通过投影器将事件流转换为面向查询的视图
    • 使用聚合索引器进行全量或增量投影

  • 命令与聚合根

    • 命令接口与基础命令为命令层提供统一结构
    • 聚合根接口定义命令处理与事件管理契约

  • 聚合根与事件存储

    • 基础聚合根提供事件收集与版本管理，事件存储负责持久化与读取

  • 聚合根与消息发布

    • 事件发布器负责将领域事件序列化并发布到事件总线

  • 投影与查询

    • 事件读取器、聚合索引器与投影器共同完成事件到视图的转换
  • 事件存储
    • 使用期望版本控制与批量保存接口降低并发冲突与写放大
    • 对大型聚合启用快照以减少事件重放成本
  • 投影
    • 全量投影按事件类型与聚合类型分批处理，结合索引器提升效率
  • 序列化
    • 事件负载采用 Base64 编码，建议在事件总线层进行压缩或限流

故障排查指南​

  • 事件 ID/类型为空或不匹配
    • 基础聚合根 AddEvent 对事件 ID、类型与聚合标识进行严格校验，若触发异常需检查命令生成的事件是否符合规范
  • 事件重复
    • 若发现相同事件 ID 导致 panic，需检查命令幂等性与事件生成逻辑
  • 序列化失败
    • 事件发布器对序列化失败进行错误包装，建议检查事件结构与编码配置
  • 事件总线处理器错误
    • 各事件总线实现对处理器返回错误采取非阻断策略，建议在日志中定位并修复处理器逻辑

Sparrow 框架通过清晰的命令接口、聚合根契约与事件存储/发布机制，为 DDD/CQRS 场景提供了可靠的命令模式实现。遵循本文的定义规范、交互流程与最佳实践，可有效保障命令处理的正确性、一致性与可扩展性，并在查询侧通过投影实现高性能读模型。
