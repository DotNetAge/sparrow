简介​

本指南围绕 Sparrow 框架的聚合根设计，系统阐述 AggregateRoot 接口的设计理念、职责边界、一致性保障与业务规则校验；详解生命周期管理（版本控制/乐观锁、事件处理、状态管理）；解释在事件溯源架构中的角色（命令处理、事件生成、状态应用、事件重播）；并提供实现模式与最佳实践，包括 HandleCommand、ApplyEvent、LoadFromEvents 等关键方法的参考路径与流程图。

项目结构​

Sparrow 在 pkg/entity 下提供了聚合根与事件、命令、快照的基础抽象，配合 usecase 包中的事件存储接口，形成事件溯源的基础设施。模板文档提供了聚合根生成与配套使用的参考示例。

  • 聚合根接口与基类
    • 接口职责：统一聚合标识、版本控制（乐观锁）、命令处理、事件生成与应用、未提交事件管理、事件重播、快照加载、业务验证。
    • 基类提供版本号、未提交事件列表、基础实体字段与常用方法。
  • 领域事件与命令
    • 事件接口定义事件标识、类型、聚合关联、时间戳、版本等；基础事件实现通用字段与工具。
    • 命令接口定义命令标识与聚合ID，确保命令与聚合的绑定一致。
  • 快照
    • 快照接口定义聚合ID、类型、版本、创建时间与状态序列化/反序列化能力。
  • 事件存储
    • 提供基础保存/读取/加载、查询扩展、快照扩展、批量扩展等能力，适配数据库型与事件流型存储。

聚合根在事件溯源中的角色：

  • 命令 -> 聚合根：聚合根根据命令生成领域事件（可能触发业务规则校验），并将事件加入未提交事件列表。
  • 事件存储：持久化事件与聚合版本，支持按版本/类型/时间范围查询。
  • 事件回放：通过 LoadFromEvents 重放事件，使聚合状态与事件流一致。
  • 快照优化：对历史较长的聚合，可通过快照加速重建。

聚合根接口与生命周期​

  • 聚合标识：GetAggregateType()/GetAggregateID() 明确聚合类型与唯一标识。
  • 版本控制：Get/Set/IncrementVersion() 实现乐观锁，结合事件存储的期望版本参数。
  • 事件管理：HandleCommand() 生成事件并应用；ApplyEvent() 将事件应用于状态；GetUncommittedEvents()/MarkEventsAsCommitted() 管理未提交事件。
  • 重播与快照：LoadFromEvents() 重放事件；LoadFromSnapshot() 从快照恢复（可选）。
  • 业务验证：Validate() 预留接口，可在聚合状态变更后执行。
  • 命令载体：Command 接口与 BaseCommand 提供命令ID与聚合ID绑定。
  • 命令到事件：聚合根的 HandleCommand(cmd) 依据业务规则生成领域事件；事件由 AddEvent() 自动设置版本并加入未提交事件列表。
  • 事件应用：ApplyEvent(event) 将事件属性映射到聚合状态；LoadFromEvents(events) 支持批量重放。
  • 事件接口：DomainEvent 定义事件标识、类型、聚合关联、时间戳与版本。
  • 基础事件：BaseEvent 提供通用字段与 Payload 解析工具。
  • 重播：LoadFromEvents(events) 逐个调用 ApplyEvent()，确保状态与事件流一致。
  • 快照：Snapshot 接口与 SnapshotData[T] 提供状态序列化/反序列化与版本管理；LoadFromSnapshot(snapshot) 可加速重建。
  • 聚合根版本：BaseAggregateRoot 维护 Version；每次 AddEvent() 递增版本并设置事件版本。
  • 事件存储期望版本：SaveEvents(ctx, aggregateID, events, expectedVersion) 用于并发冲突检测与保护。
  • 未提交事件：GetUncommittedEvents()/MarkEventsAsCommitted() 管理待持久化的事件集合。

  • 实体与聚合根：Entity 接口定义实体的标识与时间戳；AggregateRoot 继承 Entity，扩展聚合标识、版本控制与事件能力。

  • 值对象：在聚合内部可作为字段存在，但不在本节展开；聚合根负责维护其一致性边界与业务规则。

  • 命令幂等与事件去重：AddEvent() 对事件ID与聚合ID进行严格校验，避免重复事件与跨聚合事件污染。

  • 事件版本与聚合版本解耦：事件版本由聚合根在 AddEvent() 时设置，确保事件顺序与聚合状态一致。

  • 查询与事件存储分离：事件存储仅负责事件持久化与读取；查询侧通过投影或快照实现读优化。

  • 快照策略：对历史事件较多的聚合，定期生成快照并基于快照+增量事件进行重建，降低重播成本。

  • 聚合根依赖事件存储接口以实现持久化与查询；依赖命令接口以接收业务指令；依赖事件接口以生成与应用事件；依赖快照接口以优化重建。

  • 基础事件与命令提供通用字段与工具，减少重复实现。

  • 事件批量写入：EventStore 的批量能力可降低写放大与网络开销。
  • 快照重建：对历史较长的聚合，优先从快照恢复，再增量应用事件，显著降低重播成本。
  • 并发控制：通过期望版本与乐观锁避免写冲突；必要时引入重试与退避策略。
  • 事件查询：利用按版本/类型/时间范围查询，减少不必要的全量扫描。

故障排查指南​

  • 事件ID/类型为空：AddEvent() 对事件ID、事件类型与聚合ID进行严格校验，若为空将抛出异常，需检查命令与事件构造逻辑。
  • 跨聚合事件：AddEvent() 校验事件聚合ID与当前聚合ID一致，不一致将抛出异常，需检查命令绑定与事件归属。
  • 重复事件ID：AddEvent() 检测未提交事件列表中是否存在相同事件ID，重复将抛出异常，需确保命令幂等或事件ID唯一。
  • 并发冲突：SaveEvents 的期望版本不匹配会导致冲突，应进行重试或回滚处理。

Sparrow 的聚合根设计以 AggregateRoot 接口为核心，结合 BaseAggregateRoot 基类与事件、命令、快照等基础设施，提供了事件溯源所需的命令处理、事件生成与应用、版本控制与重播能力。通过严格的事件校验与乐观锁机制，保障了聚合状态的一致性与可靠性；通过快照与批量能力优化了大规模历史事件的处理性能。实际实现中，建议遵循模板与示例文档中的模式，确保命令与事件的正确绑定、事件版本与聚合版本的同步，以及快照与增量事件的协同使用。
