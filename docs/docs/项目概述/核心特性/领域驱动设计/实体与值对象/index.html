引言​

本指南围绕 Sparrow 框架中的“实体（Entity）”与“值对象（Value Object）”展开，系统阐述它们在领域驱动设计（DDD）中的角色、差异与协作方式；重点覆盖：

  • 实体的身份标识、相等性与生命周期管理
  • 值对象的不可变性、结构相等性与组合使用模式
  • ID 生成策略、相等性比较与序列化处理
  • 在事件溯源中的序列化与反序列化机制
  • 聚合根与事件、快照之间的协作关系与最佳实践

项目结构​

围绕实体与值对象的相关代码主要集中在 entity 子包与持久化仓库测试中，形成“接口定义—基础实现—聚合根—事件与快照—示例实体”的层次化结构。

  • 实体接口与基础实体

    • 实体接口定义了统一的标识与时间戳访问能力，便于仓储与业务逻辑解耦。
    • 基础实体提供标准字段与构造器，统一 ID、创建与更新时间的初始化与设置。

  • 聚合根接口与基础聚合根

    • 聚合根在实体基础上增加版本号、未提交事件列表与事件溯源相关方法。
    • 基础聚合根实现事件追加、版本递增、事件标记等关键流程。

  • 领域事件与基础事件

    • 领域事件接口定义事件的标识、类型、聚合关联与版本。
    • 基础事件实现通用字段与序列化辅助方法，支持负载的编码与解码。

  • 快照

    • 快照接口与泛型实现提供聚合状态的序列化/反序列化能力，配合事件回放加速恢复。

  • 示例实体

    • Task 与 Session 展示了如何嵌入基础实体以实现业务实体。

  • 值对象示例与测试

    • 通过测试用例展示值对象在持久化层的映射与读写。

详细组件分析​

实体与值对象的区别与应用​

  • 实体（Entity）

    • 具有唯一身份标识（ID），生命周期内可变；相等性基于标识而非内容。
    • 在 Sparrow 中通过实体接口与基础实体实现统一的标识与时间戳管理。

  • 值对象（Value Object）

    • 无唯一标识，相等性由结构组成决定；通常不可变，推荐以只读方式使用。
    • 在仓库测试中演示了值对象的组合使用与持久化映射。

  • 身份标识

    • 实体接口要求提供 GetID/SetID；基础实体提供构造器与设置器，确保创建时即具备稳定标识。

  • 相等性

    • 建议基于 ID 判断相等，避免使用值对象的结构相等性作为实体判同依据。

  • 生命周期

    • 通过 CreatedAt/UpdatedAt 字段记录创建与变更时间；实体可随业务操作更新状态。

  • 不可变性

    • 值对象应尽量保持只读，避免在外部修改其内部字段，推荐通过构造函数一次性赋值。

  • 结构相等性

    • 值对象的相等性由其组成字段决定；在比较时应逐字段对比。

  • 组合使用

    • 通过在实体中嵌入值对象，实现复杂业务数据的模块化与复用；仓库测试展示了多层嵌套值对象的映射与持久化。

  • ID 生成策略

    • 基础实体在创建时统一设置时间戳；示例实体 Task 提供独立的 ID 生成函数，建议结合业务场景选择全局唯一策略（如 UUID、雪花算法等）。

  • 相等性比较

    • 建议以实体 ID 为主，值对象以结构相等为主；避免混用导致的歧义。

  • 序列化处理

    • 值对象在持久化层通过扁平化字段映射（如 contact_info_email）实现序列化；事件负载采用 base64 编码 + JSON 解析的方式进行序列化与反序列化。

  • 事件序列化

    • 基础事件包含事件 ID、类型、聚合 ID/类型、时间戳与版本；Payload 可承载事件负载，采用 base64 编码与 JSON 解析。

  • 事件反序列化

    • 提供泛型 UnmarshalEvent 辅助方法，自动完成 base64 解码与 JSON 反序列化，便于事件处理器安全地提取负载。

  • 事件存储与回放

    • 聚合根维护未提交事件列表，保存后标记为已提交；快照用于加速大规模事件回放。

  • 协作关系

    • 聚合根负责业务规则与状态变更；通过事件记录历史；通过快照减少回放成本。

  • 最佳实践

    • 明确事件类型与聚合 ID 的一致性校验；严格禁止外部直接调用内部事件添加方法；事件版本与聚合版本同步递增；快照保存时机与版本管理清晰。

  • 模板与生成

    • 聚合根模板文档提供了生成脚本与示例，指导如何实现 ApplyEvent、LoadFromEvents、快照集成等。

  • Task

    • 嵌入基础实体，提供任务状态机与重试控制；ID 生成策略独立实现。

  • Session

    • 嵌入基础实体，提供会话数据与过期时间管理；核心业务规则以过期判断为核心。

  • 命令接口

    • 提供命令 ID 与聚合 ID 的统一访问，保证命令处理与聚合根的绑定关系。

  • 与聚合根协作

    • 聚合根通过 HandleCommand 生成领域事件，再通过 ApplyEvent 应用到状态，最终将事件加入未提交列表以便持久化。

性能考虑​

  • 事件回放优化
    • 对于历史较长的聚合，优先加载快照并回放后续事件，降低回放成本。
  • 版本与并发
    • 使用聚合版本号进行乐观锁控制，避免并发写冲突带来的状态不一致。
  • 值对象映射
    • 在持久化层采用扁平化字段映射，减少 ORM 层的复杂度与查询开销。

故障排查指南​

  • 事件一致性错误
    • 当事件 ID、事件类型或聚合 ID 为空或不匹配时，基础聚合根会抛出异常；检查命令与事件构造逻辑。
  • 重复事件 ID
    • 同一聚合内不允许出现相同事件 ID；若发生，请检查事件生成策略与幂等性设计。
  • Payload 解析失败
    • 基础事件的负载采用 base64+JSON；若解析失败，检查序列化过程与编码格式。

Sparrow 框架通过“实体接口 + 基础实体”、“聚合根接口 + 基础聚合根”、“领域事件 + 基础事件”、“快照接口 + 泛型实现”的分层设计，清晰地区分了实体与值对象的角色与职责，并在事件溯源与快照机制下提供了可扩展、可维护的领域建模能力。实践中应坚持：

  • 实体以 ID 为相等性依据，值对象以结构相等性为准
  • 事件与快照的版本管理与序列化规范
  • 聚合根内的业务规则与状态变更应通过事件记录与回放保障一致性
