简介​

本技术文档围绕混合调度器展开，系统阐述其架构设计与实现原理，重点说明如何结合并发与顺序两种调度模式的优势，实现任务路由与动态切换能力，并给出配置策略、性能优化建议及典型使用场景。混合调度器通过“任务类型”与“任务特性”（如定时、周期）双重维度，智能选择并发或顺序执行策略，从而在吞吐与一致性之间取得平衡。

项目结构​

混合调度器位于任务子系统中，核心文件包括：

  • 混合调度器：负责任务路由与模式切换
  • 并发调度器：支持多任务并发执行与限流
  • 顺序调度器：保证单协程串行执行，适合定时/周期任务
  • 接口与模型：统一的任务抽象、状态与执行模式
  • 包装器与构建器：对外暴露简洁的API与任务装配能力
  • 引导层：在应用启动时集成与初始化
  • 混合调度器（HybridScheduler）
    • 维护并发/顺序两个子调度器实例
    • 通过任务类型映射与任务特性（定时/周期）决定执行策略
    • 提供统一的生命周期与任务管理接口
  • 并发调度器（ConcurrentScheduler）
    • 多任务并发执行，支持最大并发数限制
    • 内置任务监控、过期清理与重试机制
  • 顺序调度器（SequentialScheduler）
    • 单协程串行执行，严格保证任务顺序
    • 适用于定时/周期任务，具备指数退避重试与周期调度
  • 接口与模型（TaskScheduler/Task/TaskInfo/TaskStatus/ExecutionMode）
    • 统一抽象，便于替换与扩展
  • 包装器（SchedulerWrapper）
    • 对外提供简洁API，支持并发数、任务类型模式映射等配置
  • 构建器（TaskBuilder）
    • 任务装配工具，支持类型、调度时间、周期、超时、重试等

混合调度器在启动时分别启动并发与顺序两个子调度器；在调度任务时，依据以下规则选择执行路径：

  • 若任务类型被显式映射为顺序模式，则走顺序调度器
  • 否则若任务为定时或周期任务，则走顺序调度器
  • 否则走并发调度器

混合调度器（HybridScheduler）​

  • 任务路由
    • 类型映射：通过 WithConcurrent/WithSequential 动态配置任务类型与执行模式
    • 时间特性：定时/周期任务默认顺序执行，确保时序与一致性
    • 默认策略：未匹配上述情况的任务默认并发执行
  • 生命周期与资源管理
    • Start：依次启动并发与顺序调度器
    • Stop/Close：分别停止与优雅关闭两个子调度器
  • 任务管理
    • Cancel：在两个子调度器中尝试取消，任一成功即返回
    • GetTaskStatus：优先从并发调度器查询，不存在则查询顺序调度器
    • ListTasks：合并两个子调度器的任务列表
    • SetMaxConcurrentTasks：仅对并发调度器生效
  • 并发控制
    • 最大并发数限制与活跃任务计数
    • 条件变量与轮询机制协调任务进入
  • 任务执行
    • 即时任务立即执行；定时/重试任务由监控器触发
    • 超时控制与重试策略（线性递增至上限）
  • 资源管理
    • TTL过期清理已完成/取消/死信任务
    • 最大任务数量限制，淘汰最老已完成任务
  • 串行执行
    • 单协程执行，严格顺序；待执行队列与完成通知通道协调
  • 定时/周期任务
    • 监视器周期扫描，将到期任务加入待执行队列
    • 支持指数退避重试与周期性再次调度
  • 资源管理
    • TTL过期清理；最大任务数量限制，淘汰最老已完成任务
  • TaskScheduler：统一的调度器接口，包含调度、取消、查询、列表、并发数设置、启动/停止/关闭
  • Task/TaskInfo/TaskStatus：任务实体、任务信息与状态枚举
  • ExecutionMode：执行模式（并发/顺序）
  • SchedulerWrapper
    • 对外提供 RunTask*/RunTypedTask*/CancelTask*/ListTasks 等便捷方法
    • 支持默认重试次数、最大并发数、任务类型执行模式映射
    • 默认使用混合调度器，可注入自定义调度器
  • TaskBuilder
    • 任务装配工具，支持类型、调度时间、周期、超时、重试、优先级等
  • 混合调度器依赖并发/顺序两个子调度器，二者均实现 TaskScheduler 接口
  • 包装器在创建时可注入自定义调度器或默认混合调度器，并应用并发数与任务类型映射配置
  • 引导层通过 Tasks(...) 选项将调度器集成到应用生命周期中

  • 并发控制

    • 并发调度器通过最大并发数限制与活跃任务计数控制资源占用，避免过载
    • 建议根据CPU/IO特性与下游依赖设置合理上限

  • 任务监控与清理

    • 并发/顺序调度器均内置定时清理与TTL回收，避免内存膨胀

  • 重试策略

    • 并发调度器采用线性递增重试间隔；顺序调度器采用指数退避，降低抖动

  • 任务类型映射

    • 将高时延/易冲突任务映射为顺序执行，提升稳定性与一致性

  • 任务未执行

    • 检查任务类型映射与任务特性（定时/周期），确认被路由到顺序调度器
    • 核对调度时间是否已到达

  • 任务堆积

    • 适当提高并发调度器最大并发数（注意资源限制）
    • 分析任务重试频率，优化上游依赖

  • 取消无效

    • 确认任务ID正确；顺序调度器可能因串行执行而延迟响应

  • 状态查询异常

    • 并发调度器优先查询，若不存在再查顺序调度器；确认任务是否仍在内存中（受TTL影响）

混合调度器通过“类型映射 + 任务特性”的双维度路由，实现了并发与顺序两种模式的有机融合。它既能在高吞吐场景下发挥并发优势，又能在需要时序一致性的场景中保证顺序执行。配合包装器的简洁API与引导层的无缝集成，开发者可以快速构建复杂任务调度系统，并根据业务特点灵活调整策略。

使用场景与最佳实践​

  • 高频低时延任务（如消息处理、简单计算）：映射为并发执行，提升吞吐

  • 定时/周期任务（如报表生成、数据同步）：默认顺序执行，确保时序与幂等

  • 高风险/易冲突任务（如外部系统调用）：映射为顺序执行，降低并发冲突

  • 混合策略：通过 WithConcurrent/WithSequential 动态配置任务类型映射，实现按需切换
