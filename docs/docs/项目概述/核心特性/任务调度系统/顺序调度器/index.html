简介​

本文件面向“顺序调度器”的技术文档，聚焦其串行执行机制与实现原理，涵盖任务按序排队与执行控制逻辑、队列管理、执行状态跟踪、错误处理与重试机制、以及在依赖性强任务场景下的优势与适用性。文档还提供配置选项、性能考虑、实际使用案例及与并发/混合调度模式的对比分析，帮助读者在需要严格顺序执行的场景中正确选择与使用顺序调度器。

项目结构​

顺序调度器位于任务调度模块中，与并发调度器、混合调度器共同构成统一的任务调度体系。核心文件包括：

  • 顺序调度器实现：sequential_scheduler.go
  • 并发调度器实现：concurrent_scheduler.go
  • 混合调度器实现：hybrid_scheduler.go
  • 任务调度器接口与数据模型：core.go
  • 任务构建器：builder.go
  • 调度器包装器（对外统一入口）：wrapper.go
  • 测试用例：tasks_test.go、app_scheduler_test.go
  • 顺序调度器（sequentialScheduler）：每次仅允许一个任务执行，上一任务完成后才开始下一任务；适合定时任务、周期性任务与强依赖链路。
  • 并发调度器（concurrentScheduler）：支持多任务并发执行，受最大并发数限制；适合无依赖或可并行的任务。
  • 混合调度器（HybridScheduler）：根据任务类型或调度配置选择顺序/并发执行；默认定时与周期任务走顺序模式。
  • 接口与模型（TaskScheduler、Task、TaskInfo、TaskStatus等）：统一调度器能力与任务元数据。
  • 任务构建器（TaskBuilder）：简化任务创建，支持定时、周期、超时、重试等配置。
  • 调度器包装器（SchedulerWrapper）：对外统一入口，封装调度器实例、默认重试、任务类型策略等。

顺序调度器采用“执行器 + 监视器”的双协程架构：

  • 执行器：遍历待执行队列，按序取出等待中的任务（满足调度时间），标记运行中并执行，完成后更新状态并通知。
  • 监视器：定时扫描任务集合，清理过期任务，检测定时任务到达时间并加入待执行队列，避免死锁。

顺序调度器（sequentialScheduler）​

  • 关键字段
    • 任务映射与互斥锁：tasks、taskMutex
    • 待执行队列与互斥锁：pendingTasks、pendingMutex
    • 执行状态与通知：isRunning、taskDoneChan
    • 队列与停止：taskQueue、taskQueueClosed、stopChan
    • TTL与容量：taskTTL、maxTaskCount
    • 并发等待：wg
  • 核心流程
    • Schedule：校验任务、清理超限、存储任务、入队、通知执行器
    • runTaskExecutor：按序取出任务、标记运行、执行、更新状态、通知完成
    • runTaskMonitor：定时清理过期任务、检测定时任务到达并入队
    • Cancel：调用取消回调、更新状态、从待执行队列移除
    • 状态查询与列举：基于TaskInfoProvider接口
  • 顺序保障
    • isRunning确保同一时刻仅有一个任务在执行
    • pendingTasks按插入顺序维护，执行器按序取出
    • taskDoneChan用于唤醒/阻塞，避免忙等
  • 错误处理与重试
    • 执行失败时根据MaxRetries/RetryCount进行指数退避重试
    • 达到最大重试次数后标记失败
  • 资源清理
    • cleanupExpiredTasks：清理已完成/取消/失败且超过TTL的任务
    • cleanupOldTasks：当任务数量超过maxTaskCount时，删除最旧的已完成任务

  • 关键字段

    • 任务映射与互斥锁：tasks、taskMutex
    • 并发控制：maxConcurrent、activeTasks、activeTasksCond
    • 停止信号：stopChan
    • TTL与容量：taskTTL、maxTaskCount

  • 核心流程

    • Schedule：即时执行或入队，若到达执行时间则异步执行
    • runTaskMonitor：定时轮询，处理定时/重试任务
    • executeTask：并发槽位控制、超时、重试/死信处理

  • 并发保障

    • activeTasks计数与条件变量控制并发上限

  • 顺序对比

    • 无全局顺序约束，任务可能并发执行
    • 适合无依赖或可并行的任务

  • 设计目标

    • 根据任务类型或调度配置选择顺序/并发执行
    • 默认定时与周期任务走顺序模式，其他走并发模式

  • 核心流程

    • WithSequential/WithConcurrent：配置任务类型到执行模式映射
    • Schedule：优先按类型映射，其次按调度时间/周期特性判断
    • Cancel/GetTaskStatus/ListTasks：在两个子调度器间协调

  • TaskScheduler：统一的调度器接口，包含Schedule、Cancel、GetTaskStatus、ListTasks、SetMaxConcurrentTasks、Start、Stop、Close

  • Task：任务接口，包含ID、Type、Schedule、Handler、OnComplete、OnCancel、IsRecurring、GetInterval、GetTimeout

  • TaskInfo：任务元数据，包含状态、调度时间、创建/更新时间、重试计数、错误信息、TTL等

  • TaskStatus：任务状态枚举（waiting/running/completed/cancelled/failed/retrying/dead_letter）

  • 功能

    • WithID、WithType、WithHandler、WithOnComplete、WithOnCancel
    • WithSchedule、WithRecurring、WithTimeout、WithRetry、WithPriority
    • Build：生成内置任务实现builtTask，实现Task与TaskInfoProvider接口

  • 用途

    • 简化任务创建，统一设置调度时间、周期、超时、重试等

  • 功能

    • 统一入口：RunTask、RunTaskWithID、RunTaskAt、RunTaskWithIDAt、RunTaskRecurring、RunTaskWithIDRecurring、RunTaskAfter、RunTaskAfterWithID、RunTypedTask、RunTypedTaskWithID
    • CancelTask、GetTaskStatus、ListTasks、SetTaskTypeMode、SetMaxConcurrentTasks、Stop、Close
    • 默认重试次数配置、任务类型执行模式映射、并发数设置

  • 与调度器的关系

    • 默认使用混合调度器；可注入自定义调度器实例
    • 对并发调度器与混合调度器内的并发部分分别设置最大并发数

  • 顺序调度器依赖

    • Task接口与TaskInfoProvider接口以获取任务状态与调度信息
    • builtTask实现Task与TaskInfoProvider，提供调度时间、周期、超时、重试等能力
    • 通过channels与mutexes协调执行器与监视器

  • 并发调度器依赖

    • 条件变量与互斥锁控制并发槽位
    • 定时器轮询处理定时/重试任务

  • 混合调度器依赖

    • 组合顺序与并发调度器，按任务类型或调度特性选择执行路径

  • 顺序执行的吞吐量

    • 顺序调度器每次仅执行一个任务，吞吐量受限于单任务执行时间
    • 适合强依赖链路与串行化资源访问场景

  • 并发执行的吞吐量

    • 并发调度器受maxConcurrent限制，可通过SetMaxConcurrentTasks调整
    • 适合无依赖或可并行的任务

  • 混合调度器的选择

    • 默认定时/周期任务走顺序，其他走并发，兼顾吞吐与顺序需求
    • 可通过WithSequential/WithConcurrent精确控制任务类型执行模式

  • 内存与资源

    • 任务TTL与最大数量限制防止内存膨胀
    • cleanupExpiredTasks与cleanupOldTasks定期清理过期/旧任务

  • 重试与退避

    • 顺序调度器采用指数退避，最大间隔限制，避免频繁重试导致资源争用

  • 通道与锁

    • 使用带缓冲队列与通知通道，减少忙等与上下文切换
    • 互斥锁保护关键数据结构，避免竞态

  • 任务未执行

    • 检查任务调度时间是否已到；顺序调度器仅在到达时间后执行
    • 确认任务状态是否为waiting；查看监视器是否正常扫描

  • 任务卡住

    • 检查isRunning是否为true且无通知；确认执行器循环是否被阻塞
    • 查看taskDoneChan是否被消费；避免通道满导致通知丢失

  • 任务取消无效

    • 确认任务是否仍在待执行队列中；取消后应从队列移除
    • 检查OnCancel回调是否正确设置

  • 重试异常

    • 检查MaxRetries与RetryCount；确认NextRetryAt是否正确设置
    • 观察指数退避是否达到最大间隔

  • 并发问题

    • 并发调度器需关注maxConcurrent设置；过大可能导致资源争用
    • 使用条件变量与互斥锁确保并发安全

顺序调度器通过严格的串行执行与完善的队列管理，确保任务在强依赖与串行化资源访问场景下的顺序性与一致性。结合监视器的定时扫描与清理机制，能够稳定处理定时/周期任务与过期任务。对于需要严格顺序执行的业务，顺序调度器是首选；在混合场景下，混合调度器可根据任务类型灵活选择执行模式，兼顾吞吐与顺序需求。

配置选项与使用建议​

  • 顺序调度器

    • 任务TTL：控制已完成/取消/失败任务的保留时间
    • 最大任务数量：控制内存占用，超过阈值时清理最旧任务
    • 待执行队列：维护任务执行顺序

  • 并发调度器

    • 最大并发数：通过SetMaxConcurrentTasks设置
    • 重试策略：支持最大重试次数与线性退避

  • 混合调度器

    • 任务类型映射：WithSequential/WithConcurrent
    • 默认行为：定时/周期任务走顺序，其他走并发

  • 任务构建器

    • 支持定时、周期、超时、重试、优先级等配置

  • 包装器

    • 默认重试次数、并发数设置、任务类型策略

  • 顺序执行案例

    • 顺序调度器测试：验证任务按提交顺序执行
    • 应用层测试：定时任务与周期任务按顺序执行

  • 并发执行案例

    • 并发调度器测试：验证最大并发数限制
    • 应用层测试：并发任务几乎同时开始执行

  • 混合调度器案例

    • 应用层测试：不同任务类型按策略执行（并发/顺序）

  • 对比分析

    • 顺序调度器：吞吐低但顺序强，适合强依赖与串行化资源
    • 并发调度器：吞吐高但无全局顺序，适合无依赖或可并行任务
    • 混合调度器：兼顾两者，按任务类型与调度特性选择模式
