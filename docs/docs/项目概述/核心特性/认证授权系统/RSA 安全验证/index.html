简介​

本文件系统性阐述项目中的 RSA 安全验证机制，重点覆盖以下方面：

  • RSA 公钥验证在认证授权中的应用场景与安全优势
  • 公钥生成、存储与分发的安全实现（密钥对创建与管理流程）
  • RSA 中间件的实现原理（公钥验证、签名验证、安全策略执行）
  • RSA 权限验证的具体实现（请求签名的生成与验证流程）
  • 安全最佳实践（密钥轮换、安全存储与传输加密）
  • 完整的配置与使用示例路径，以及与现有认证授权流程的集成方式

项目结构​

围绕 RSA 安全验证的相关模块主要分布在以下位置：

  • 认证与授权核心：pkg/auth（接口、实现、错误、令牌模型与生成器）
  • HTTP 中间件：pkg/adapter/http/middlewares（RSA 客户端认证中间件与 JWT 中间件）
  • 引导与装配：pkg/bootstrap（授权装配与中间件注册）
  • 配置：pkg/config（应用配置结构）
  • 接口层（认证与授权核心）
    • NonceStore：随机数存储与去重，防重放攻击
    • PublicKeyProvider：按客户端 ID 提供 RSA 公钥
    • SignatureCache：签名验证结果缓存，提升性能
    • RSAAuthConfig：RSA 认证配置（请求头名称、时间窗口、各组件注入）
  • 内存实现（认证与授权核心）
    • MemoryNonceStore：基于内存的随机数存储与定期清理
    • MemoryPublicKeyProvider：基于内存的公钥映射与并发安全
    • MemorySignatureCache：基于内存的签名验证结果缓存与定期清理
  • HTTP 中间件（RSA 客户端认证）
    • RSAClientAuthMiddleware / ConfigurableRSAClientAuthMiddleware：从请求头读取必要信息，校验时间戳、随机数、签名，写入上下文
    • NewRSAClientAuthMiddleware / NewConfigurableRSAClientAuthMiddlewareFromPEM：从 PEM 字符串解析公钥并创建中间件
  • 令牌与声明（JWT）
    • UserClaims：扩展的 JWT 声明（含用户 ID、角色等）
    • Token：访问令牌、刷新令牌、作用域与过期时间
    • TokenGenerator：令牌生成与刷新
  • 引导与装配
    • Authorization：集中注册与获取中间件

RSA 安全验证在 HTTP 请求链路中的工作流如下：

RSA 接口与配置​

  • 接口职责清晰，便于替换实现（例如持久化存储、分布式缓存）
  • 配置项支持灵活定制请求头名称与时间窗口，满足不同部署场景
  • MemoryNonceStore：并发安全的内存存储，定时清理过期随机数
  • MemoryPublicKeyProvider：支持动态添加公钥，按 clientID 查询
  • MemorySignatureCache：缓存验证结果，定时清理过期条目
  • 功能要点
    • 从请求头读取 clientID、signature、nonce、timestamp
    • 校验时间戳是否在允许的时间窗口内
    • 检查随机数是否重复（NonceStore）
    • 通过 PublicKeyProvider 获取公钥
    • 读取请求体并拼接数据，Base64 解码签名后使用 SHA256+RSA-PSS 验证
    • 成功后记录随机数并缓存验证结果
    • 将 clientID 写入上下文，继续后续处理
  • 工厂函数
    • RSAClientAuthMiddleware：直接传入 clientID->公钥映射
    • NewRSAClientAuthMiddleware：从 PEM 字符串构建公钥映射
    • NewConfigurableRSAClientAuthMiddlewareFromPEM：结合默认配置与 PEM 公钥
  • 支持从 PEM 字符串解析多种格式的公钥（PKIX、SSH），增强兼容性
  • 提供工厂函数将 PEM 映射转换为中间件所需的公钥集合
  • JWT 中间件负责基于 HMAC 的令牌解析与验证，并将用户上下文写入请求
  • RSA 中间件与 JWT 中间件可并行使用：前者保护 API 调用签名，后者保护会话/令牌

  • 正常流程：签名正确、时间戳在窗口内、随机数唯一

  • 缺失头、无效签名、过期时间戳、无效 clientID、重放攻击等边界条件均有覆盖

  • 中间件依赖接口层（NonceStore、PublicKeyProvider、SignatureCache），通过组合注入实现可替换性

  • 内存实现作为默认实现，便于快速启动与演示

  • 引导层（Authorization）统一注册与获取中间件，便于在应用启动阶段装配

  • 签名缓存（SignatureCache）：显著降低重复请求的验签开销

  • 随机数缓存（NonceStore）：避免重复请求被判定为重放攻击

  • 定时清理：内存实现通过周期性清理减少内存占用

  • 并发安全：读写锁保障高并发场景下的稳定性

  • 常见错误与定位

    • 缺少必要请求头：检查客户端是否正确设置 X-Client-ID、X-Client-Signature、X-Nonce、X-Timestamp
    • 无效签名：确认签名算法为 SHA256+RSA-PSS，签名数据顺序为 clientID + nonce + timestamp + body
    • 时间戳过期：调整客户端时间同步，或增大 TimeWindow
    • 重复请求：确保每次请求使用唯一 nonce
    • 未授权的客户端：确认 clientID 是否存在于 PublicKeyProvider

  • 错误码与含义

    • 未授权的客户端、缺少必要的客户端验证信息、无效的签名、无效的时间戳、时间戳已过期、无效的随机数、重复的请求、读取请求体失败、请求验证失败

该 RSA 安全验证机制通过明确的接口设计与内存实现，提供了可插拔、可扩展的客户端签名验证能力。其核心特性包括：

  • 基于时间戳与随机数的防重放策略
  • SHA256+RSA-PSS 的强健签名验证
  • 缓存与定时清理机制提升性能与资源利用率
  • 与 JWT 中间件协同，适配多层安全需求

建议在生产环境中结合持久化存储与分布式缓存替换内存实现，并配合严格的密钥轮换与传输加密策略。

密钥生成与管理流程（概念性说明）​

  • 生成密钥对：使用安全的随机源生成 RSA 私钥与公钥
  • 分发公钥：通过受控渠道将公钥发布给客户端；建议采用证书链或受信分发平台
  • 存储私钥：仅在受信环境中保存私钥，限制访问权限与审计日志
  • 密钥轮换：定期更换密钥，平滑过渡期间保留旧公钥以兼容历史请求

集成到现有认证授权流程（步骤指引）​

  • 在引导层注册中间件
    • 使用 Authorization.AddMiddleware 或 AddMiddlewares 注册 RSA 中间件
    • 如需与 JWT 协同，同时注册 JWTAuthMiddleware
  • 配置公钥提供者
    • 使用 NewMemoryPublicKeyProvider 注入 clientID->公钥映射
    • 生产环境建议替换为持久化实现
  • 配置时间窗口与请求头
    • 通过 DefaultRSAAuthConfig 自定义请求头名称与时间窗口
  • 客户端签名生成
    • 按约定顺序拼接数据：clientID + nonce + timestamp + body
    • 使用 SHA256+RSA-PSS 对摘要进行签名并 Base64 编码
    • 设置请求头：X-Client-ID、X-Client-Signature、X-Nonce、X-Timestamp
