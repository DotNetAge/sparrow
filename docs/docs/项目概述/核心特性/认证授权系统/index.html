简介​

本文件面向 Sparrow 框架的认证授权系统，围绕以下目标展开：

  • 基于 JWT 的认证机制：令牌生成、验证与刷新，涵盖 HS256 签名与声明结构。
  • 基于 Casbin 的 RBAC 权限控制：中间件集成、模型与策略路径配置。
  • 中间件体系：JWT 认证中间件、RSA 客户端签名中间件、Casbin RBAC 中间件。
  • 安全最佳实践：防重放、时间戳校验、签名缓存、错误分类与响应。
  • 使用示例：如何在应用中配置与启用认证授权功能。

项目结构​

认证授权相关代码主要分布在如下模块：

  • auth：声明、主体、令牌生成与 RSA 配置及实现
  • bootstrap：授权装配与中间件注册
  • adapter/http/middlewares：JWT、RSA、Casbin 中间件
  • config：Casbin 配置项
  • tests：JWT 与 RSA 中间件的单元测试
  • 用户声明(UserClaims)：扩展标准注册声明，包含用户标识与角色列表，用于承载主体身份与权限元信息。
  • 主体(Principal)：在请求上下文中传递当前用户的身份信息（用户名、用户ID、角色）。
  • 令牌(Token)：封装访问令牌、刷新令牌、授权范围与过期间隔。
  • 令牌生成器(TokenGenerator)：负责生成 JWT 令牌、刷新令牌，以及基于现有刷新令牌发放新令牌。
  • RSA 安全中间件：基于 RSA-PSS 签名的客户端认证，具备时间戳校验、随机数防重放、签名缓存等能力。
  • Casbin RBAC 中间件：基于用户名、资源路径与动作进行权限强制检查。
  • 引导装配(Authorization)：集中注册与管理中间件，供 App 启动时挂载。

JWT 令牌生成与验证​

  • 令牌生成器
    • 支持生成访问令牌与刷新令牌，分别设置不同过期时间。
    • 使用 HS256 对自定义声明进行签名。
  • 令牌验证
    • 中间件从 Authorization 头解析 Bearer 令牌。
    • 使用 UserClaims 解析并校验签名方法、过期、未生效、签名无效等场景。
    • 成功后将 Principal 写入请求上下文，供后续中间件与处理器使用。
  • 使用刷新令牌调用 RefreshToken 接口。
  • 解析并校验刷新令牌有效性。
  • 更新签发时间为当前时间，重新生成新的访问令牌与刷新令牌。
  • 返回新的 Token 结构体。
  • 声明(UserClaims)
    • 继承标准注册声明，新增用户ID与角色数组。
    • 提供构造函数统一设置 Issuer、Subject、过期与签发时间。
  • 主体(Principal)
    • 在上下文中以 UserKey 作为键存储与检索。
    • 便于业务层通过 context 获取当前用户身份。
  • 核心流程
    • 从请求头读取客户端ID、签名、随机数、时间戳。
    • 校验时间戳是否在允许窗口内。
    • 检查随机数是否已存在（防重放），并写入存储。
    • 依据客户端ID获取公钥，解码签名并使用 SHA256+RSA-PSS 验证。
    • 可选签名缓存，提升性能。
  • 配置
    • 默认请求头、时间窗口、以及可注入的 NonceStore、PublicKeyProvider、SignatureCache。
  • 错误处理
    • 缺失信息、时间戳无效、签名无效、客户端不存在、重复请求、读取请求体失败、验证失败等。
  • 中间件职责
    • 从上下文获取用户名（需前置 JWT 中间件）。
    • 以请求路径为资源、HTTP 方法为动作，调用 Enforce 执行策略检查。
    • 通过则放行，否则返回 403。
  • 配置
    • 通过 CasbinConfig 指定模型与策略文件路径，供 Casbin 初始化使用。
  • Authorization
    • 负责注册与命名中间件，供 App 启动时挂载。
  • App
    • 通过 Bootstrap 初始化 Gin 引擎与中间件链路。
  • 组件耦合
    • JWT 中间件依赖 UserClaims 与 Principal，用于解析与上下文传递。
    • RSA 中间件依赖 RSAAuthConfig 与若干接口（NonceStore、PublicKeyProvider、SignatureCache）。
    • RBAC 中间件依赖 Casbin Enforcer 与上下文中的用户名。
    • TokenGenerator 依赖 UserClaims 与 Token 结构。
  • 外部依赖
    • JWT 解析与签名：github.com/golang-jwt/jwt/v5
    • HTTP 框架：github.com/gin-gonic/gin
    • 权限引擎：github.com/casbin/casbin/v2
    • 加密库：crypto/、encoding/、golang.org/x/crypto/ssh

  • 签名缓存

    • MemorySignatureCache 通过缓存验证结果减少重复计算，建议在高并发场景启用。

  • 随机数清理

    • MemoryNonceStore 与 MemorySignatureCache 周期性清理过期条目，避免内存膨胀。

  • 时间窗口

    • 合理设置时间窗口，平衡安全性与时效性。

  • 请求体复用

    • 中间件读取请求体后重置，避免下游处理再次读取失败。

  • JWT 常见问题

    • 缺少 Authorization 头或格式错误：返回 401。
    • 令牌格式不正确、过期、未生效、签名无效：根据具体错误返回相应提示。
    • 中断请求后仍访问用户上下文：确认中间件顺序与是否正确写入上下文。

  • RSA 常见问题

    • 缺失请求头：返回“缺少必要的客户端验证信息”。
    • 时间戳过期：返回“时间戳已过期”。
    • 重复请求（重放攻击）：返回“重复的请求”，检查 NonceStore 是否正确写入。
    • 公钥解析失败：确认 PEM/SSH 格式与客户端ID匹配。

  • RBAC 常见问题

    • 上下文无用户名：确认前置 JWT 中间件已执行。
    • 策略未覆盖：检查模型与策略文件路径与规则。

Sparrow 框架的认证授权体系以 JWT 为核心，结合 RSA 客户端签名与 Casbin RBAC，形成从“身份认证”到“权限控制”的完整链路。通过清晰的接口抽象与中间件化设计，既保证了易用性，也便于扩展与测试。建议在生产环境中启用签名缓存、严格配置时间窗口与策略文件路径，并配合完善的日志与监控以保障安全与稳定性。

使用示例（步骤说明）​

  • 启用 JWT 认证中间件
    • 在路由上挂载 JWTAuthMiddleware，并传入密钥。
    • 示例参考：jwt.go
  • 启用 RSA 客户端签名中间件
    • 使用 RSAClientAuthMiddleware 或 NewConfigurableRSAClientAuthMiddlewareFromPEM。
    • 配置公钥映射、随机数存储、签名缓存与时间窗口。
    • 示例参考：rsa_perm.go、rsa_perm.go
  • 启用 Casbin RBAC 中间件
    • 在路由上挂载 RBACMiddleware，并传入已初始化的 Casbin Enforcer。
    • 示例参考：casbin.go
  • 生成与刷新令牌
    • 使用 NewTokenGenerator 创建生成器，调用 GrantToken/GrantJWTToken/RefreshToken。
    • 示例参考：gen.go、gen.go、gen.go
  • 配置 Casbin
    • 在配置中设置 model_path 与 policy_path。
    • 示例参考：casbin.go
