简介​

本文件系统性阐述 Sparrow 框架的多数据源支持体系，覆盖内存存储、Redis 缓存、PostgreSQL 关系数据库、MongoDB 文档数据库与 BadgerDB 键值数据库五类后端。文档重点说明：

  • 仓储模式（Repository Pattern）的抽象接口设计与通用基类实现
  • 各数据源的特性、适用场景与性能表现
  • 配置方法与使用示例
  • 数据源切换机制与最佳实践
  • 事务管理、连接池配置与性能优化策略
  • 如何依据业务需求选择合适的存储方案

项目结构​

围绕多数据源的核心代码主要分布在以下模块：

  • usecase 层：定义仓储接口与通用查询条件、排序等工具
  • persistence/repo 层：各数据源的仓储实现
  • bootstrap 层：应用容器、数据库连接聚合与启动流程
  • config 层：各类数据源的配置结构与默认值

  • 仓储接口与通用基类

    • 仓储接口定义了统一的 CRUD、批量操作、分页、条件查询、随机读取等能力，保证不同数据源的使用一致性。
    • 通用基类提供默认实现与错误处理骨架，具体仓储只需实现核心逻辑。

  • 数据源仓储实现

    • 内存仓储：适合测试与临时数据，提供并发安全的内存存储。
    • Redis 仓储：基于键值存储，支持 TTL、批量操作与全表扫描条件查询。
    • PostgreSQL 仓储：关系型数据库，支持软删除、事务、批量操作与复杂条件查询。
    • MongoDB 仓储：文档数据库，支持聚合与随机采样，条件查询丰富。
    • BadgerDB 仓储：嵌入式键值数据库，提供本地持久化与高效迭代。

  • 应用容器与数据库聚合

    • Container 提供依赖注入与实例缓存，支持具名与匿名解析。
    • Database 聚合 Redis、SQL、Badger 等连接，便于集中管理。

  • 配置系统

    • Config 提供应用、服务器、日志、NATS、SQL、Redis、Badger、RabbitMQ、Kafka、Casbin 等配置项默认值与结构。
    • 各数据源配置结构独立，通过 viper 加载与校验。

多数据源架构采用“统一接口 + 多实现”的模式，通过 IoC 容器按需注入具体仓储实现。应用启动时加载配置，初始化数据库连接，注册仓储提供者，运行时通过具名解析获取对应仓储实例。

仓储接口与通用基类​

  • 接口能力
    • 基本 CRUD：Save、FindByID、FindAll、Update、Delete、Exists
    • 批量操作：SaveBatch、FindByIDs、DeleteBatch
    • 分页与统计：FindWithPagination、Count、CountByField、FindByFieldWithPagination
    • 条件查询：FindWithConditions、CountWithConditions
    • 随机读取：Random
  • 基类职责
    • 提供默认实现与错误处理骨架
    • 统一 ID 校验、时间戳设置等通用逻辑
    • 提供查询条件构建器与排序字段辅助函数
  • 特点
    • 基于 map 的并发安全存储，读写性能高
    • 支持全量扫描、分页、条件查询（反射实现）
    • 适合测试、临时数据与低延迟场景
  • 并发控制
    • 读写锁保护 map，避免竞态
  • 查询与分页
    • 全表扫描实现条件查询与分页，注意大数据量下的性能开销
  • 适用场景
    • 单机测试、缓存层、临时会话数据
  • 特点
    • 键值存储，支持 TTL、批量 Pipeline、MGET/MSET
    • 全表扫描实现条件查询，不适合大规模条件过滤
    • 适合缓存、会话、事件存储等场景
  • 键空间组织
    • 通过前缀区分实体类型，键格式为 prefix:id
  • 批量与事务
    • Pipeline 提升批量写入吞吐
    • TTL 设置与查询支持
  • 适用场景
    • 高频读写、短期缓存、事件总线
  • 特点
    • 关系型数据库，支持软删除、事务、批量操作、复杂条件查询
    • 通过 sqlx 访问，支持反射扫描与自定义类型处理
    • 适合强一致性、复杂查询与事务场景
  • 事务与批量
    • SaveBatch 在单事务内执行，保证原子性
    • 支持 existsInTx、insertInTx、updateInTx 等事务内操作
  • 查询能力
    • FindByField、FindByIDs、FindWithPagination、Count、Random 等
    • 条件查询通过构建 WHERE 与排序，支持 LIMIT/OFFSET
  • 适用场景
    • 用户、订单、财务等核心业务数据
  • 特点
    • 文档数据库，支持丰富的查询操作与聚合
    • 集合命名默认基于实体类型，支持 ObjectID 与自定义 ID
    • 支持随机采样（$sample）、条件查询、分页与排序
  • 查询与聚合
    • buildMongoFilter 将 QueryCondition 映射为 BSON 过滤器
    • FindWithConditions 支持排序与分页
  • 适用场景
    • 日志、画像、内容管理、事件溯源等半结构化数据
  • 特点
    • 嵌入式键值数据库，本地持久化、迭代友好
    • 不支持软删除，删除为硬删除
    • 全表扫描实现条件查询，适合小中型数据与本地缓存
  • 键空间与迭代
    • 前缀迭代实现分页与统计
    • JSON 序列化存储实体
  • 适用场景
    • 本地缓存、事件存储、轻量级持久化
  • 组件耦合
    • 仓储实现严格依赖 usecase 接口与基类，降低耦合度
    • 各仓储对具体数据源客户端（Redis、sqlx、mongo、badger）有直接依赖
  • 容器与注册
    • Container 支持具名/匿名提供者注册与解析，便于切换数据源
  • 配置与连接
    • Database 聚合多数据源连接，App 启动时加载配置并初始化
  • 内存仓储
    • 优点：极低延迟、并发安全
    • 缺点：无持久化、容量受限
    • 优化：合理使用分页与条件过滤，避免全量扫描
  • Redis 仓储
    • 优点：高吞吐、支持 TTL、Pipeline 批量
    • 优化：合理设置 TTL、批量写入、避免全表扫描条件查询
  • PostgreSQL 仓储
    • 优点：强一致、事务、复杂查询
    • 优化：索引设计、批量操作、连接池配置、分页查询
  • MongoDB 仓储
    • 优点：灵活文档模型、聚合能力强
    • 优化：建立必要索引、合理使用投影与分页
  • BadgerDB 仓储
    • 优点：本地持久化、迭代友好
    • 优化：合理前缀、避免全表扫描、控制序列化大小

故障排除指南​

  • 常见错误类型
    • RepositoryError：仓储操作错误，包含实体类型、操作、ID、消息等信息
    • 连接失败：检查配置项（主机、端口、密码、数据库名）
    • 空 ID：调用 Save/Update/Delete 前确保实体 ID 非空
  • 排查步骤
    • 核对配置文件与环境变量
    • 检查数据源可达性与认证信息
    • 观察日志输出，定位具体仓储方法与错误原因
    • 针对条件查询，确认字段名与操作符正确

Sparrow 框架通过统一的仓储接口与多实现的数据源体系，提供了灵活、可扩展且高性能的数据访问层。开发者可根据业务需求在内存、Redis、PostgreSQL、MongoDB 与 BadgerDB 之间自由切换，并借助 IoC 容器与配置系统快速集成与部署。建议在生产环境中优先考虑一致性与扩展性，结合业务特征选择合适的数据源并实施相应的性能优化与监控策略。
