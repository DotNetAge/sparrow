简介​

本文件面向 Sparrow 框架的内存仓储实现，系统性阐述 MemoryRepository 泛型仓储的设计原理与实现细节，重点覆盖：

  • 基于 sync.RWMutex 的并发安全机制
  • 内存中的实体存储结构与 CRUD 完整实现
  • 适用场景（测试环境、临时数据存储、高性能缓存）
  • 错误处理机制、并发访问控制与最佳实践
  • 与其他数据源仓储的差异与优势

项目结构​

内存仓储位于持久化层，与实体层、用例层、错误层协同工作，形成清晰的分层职责：

  • 实体层：定义统一的 Entity 接口与基础实体结构
  • 用例层：定义仓储接口与通用查询条件/排序模型
  • 错误层：统一的仓储错误类型
  • 持久化层：MemoryRepository 提供内存中的完整 CRUD 与查询能力
  • MemoryRepository[T]: 泛型内存仓储，封装并发锁与内存映射表，提供完整的 CRUD 与查询能力
  • Repository[T] 接口：统一的仓储契约，定义所有对外操作
  • QueryOptions/QueryCondition：多条件查询与排序的基础模型
  • Entity 接口与 BaseEntity：实体抽象与标准字段（ID、创建/更新时间）

MemoryRepository 通过组合 usecase.BaseRepository[T]，在内存中维护一个 map[string]T 的实体集合，并以 RWMutex 控制并发读写。所有操作均遵循仓储接口契约，保证与上层用例解耦。

并发安全与锁策略​

  • 读写分离：使用 sync.RWMutex，在读取路径采用 RLock/RUnlock，写入路径采用 Lock/Unlock，提升并发读性能

  • 并发插入保护：insert 与 Save 在写入前双重检查，避免竞态导致的重复键冲突

  • 读写一致性：所有读取操作在进入时加读锁，退出时释放；写入操作在进入时加写锁，退出时释放

  • 存储容器：map[string]T，键为实体 ID，值为实体对象

  • 类型信息：构造时记录实体类型字符串，用于错误报告

  • 时间字段自动填充：通过反射设置 CreatedAt/UpdatedAt 字段（若实体具备对应字段）

保存（Save）​

  • 若实体 ID 为空，返回仓储错误

  • 先 Exists 检查，存在则 Update，否则 insert

  • insert 中对重复键进行二次校验，确保并发安全

  • 加写锁，检查重复键

  • 自动设置 CreatedAt/UpdatedAt（反射）

  • 写入 map

  • ID 校验，不存在返回错误

  • 加写锁，检查实体存在性

  • 自动设置 UpdatedAt（反射）

  • 更新 map

  • FindByID：空 ID 校验，读锁，返回实体或错误

  • FindAll：读锁，遍历 map 返回全部实体

  • Exists：读锁，判断键存在

  • Count：读锁，返回长度

  • Delete：空 ID 校验，加写锁，删除键

  • DeleteBatch：空 ID 列表校验，加写锁，逐个删除

  • SaveBatch：空列表直接返回；逐项设置时间字段；写锁内批量写入

  • FindByIDs：空 ID 列表返回空切片；读锁，过滤存在项

  • DeleteBatch：空列表直接返回；写锁，逐个删除

  • FindWithPagination：读锁，准备全量切片，处理默认 limit/offset，返回子切片

  • Count：读锁，返回长度

  • FindWithConditions：读锁，遍历实体，逐条件匹配（反射字段访问与类型比较）

  • CountWithConditions：读锁，同上逻辑仅计数

  • matchCondition：支持 EQ/NE/GT/LT/GTE/LTE 等操作符，含数值与字符串/时间类型比较

  • Random：读锁，收集所有 ID，随机打乱，取前 take 个，返回实体

  • Clear：写锁，重建空 map

  • 反射用于自动设置时间字段与字段值读取

  • 对指针字段做解引用处理

  • 类型不一致时进行有限的数值类型转换（int/float），字符串/布尔直比较

  • RepositoryError：统一的仓储错误类型，包含实体类型、操作、ID、消息与原因

  • 各操作在输入校验失败或未找到实体时返回该错误

  • 上层可基于错误类型进行差异化处理

  • MemoryRepository[T] 实现 usecase.Repository[T] 接口
  • 组合 usecase.BaseRepository[T]，继承通用查询模型
  • 依赖 entity.Entity 接口与 BaseEntity 标准字段
  • 依赖 errs.RepositoryError 进行错误包装
  • 测试覆盖了典型 CRUD、批量操作、分页、条件查询、并发插入与随机取样

  • 读多写少场景：RWMutex 显著提升并发读吞吐

  • 内存访问：O(1) 键查找，O(n) 遍历（分页/条件查询）

  • 反射成本：字段读取与类型比较带来一定开销，适合小中规模数据

  • 批量操作：单次加锁批量写入，减少锁竞争

  • 分页与条件查询：全量扫描，建议结合业务规模评估

  • 输入校验失败：ID 为空、字段名为空、实体 ID 为空等，均会返回 RepositoryError

  • 未找到实体：FindByID/Update/Delete/Exists 等在实体不存在时返回错误

  • 并发冲突：重复插入可能触发“实体已存在”错误，需检查 ID 生成策略

  • 类型不匹配：条件查询中类型不一致时，仅支持有限的数值类型转换

  • 随机取样：take<=0 时默认取 1 个

MemoryRepository 以简洁的内存映射表与 RWMutex 并发控制，提供了完整的仓储能力，特别适用于测试、临时数据与高性能缓存场景。其泛型设计与统一接口契约，使其易于替换与扩展。在实际应用中，应结合业务数据规模与并发特征，合理评估反射与全量扫描带来的性能影响，并在需要持久化与高可用的生产环境中选择更合适的持久化仓储实现。

适用场景​

  • 测试环境：快速搭建内存仓储，隔离外部依赖
  • 临时数据存储：短期缓存、会话数据、中间计算结果
  • 高性能缓存：热点数据的只读/短生命周期缓存

使用示例（步骤说明）​

  • 创建实体：以 Task 为例，确保其实现 Entity 接口并嵌入 BaseEntity

  • 创建仓储：调用 NewMemoryRepository*entity.Task 获取仓储实例

  • 基本操作：Save/FindByID/FindAll/Update/Delete/Count

  • 批量操作：SaveBatch/FindByIDs/DeleteBatch

  • 分页与统计：FindWithPagination/Count

  • 条件查询：FindWithConditions/CountWithConditions

  • 随机取样：Random

  • 清空（测试）：Clear

  • ID 生成：确保实体 ID 唯一且非空，避免并发插入冲突

  • 时间字段：实体可包含 CreatedAt/UpdatedAt 字段，仓储会自动设置

  • 并发控制：读多写少场景优先使用 FindByID/FindAll 等读操作

  • 条件查询：尽量缩小查询范围，避免全量扫描

  • 批量操作：合并多次写入为 SaveBatch，减少锁竞争

  • 测试清理：使用 Clear 清空测试数据，避免跨用例污染

  • 错误处理：区分 RepositoryError 与业务错误，按需降级或重试
