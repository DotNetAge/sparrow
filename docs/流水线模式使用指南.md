# 流水线模式使用指南

## 1. 概述

流水线模式（Pipeline）是 Sparrow 任务系统提供的一种特殊执行策略，适用于需要严格按照顺序执行的多阶段处理任务。通过流水线模式，可以确保相关任务按照提交顺序依次执行，避免并发执行可能带来的数据一致性问题。

## 2. 流水线模式的特点

### 2.1 核心特性

- **严格顺序执行**：任务按照提交顺序依次执行，一个任务完成后才会执行下一个任务
- **单一工作协程**：流水线模式使用单个工作协程处理所有任务
- **最大并发限制为1**：同一时间只能执行一个流水线任务
- **保证任务执行顺序**：适合对顺序有严格要求的业务场景

### 2.2 内部实现原理

流水线模式在实现上是通过以下方式保证顺序执行的：

```go
// 流水线调度器初始化
scheduler.pipelineScheduler = NewMemoryTaskScheduler(
    WithLogger(config.Logger),
    WithWorkerCount(1),  // 流水线执行只需要1个工作协程
    WithMaxConcurrentTasks(1), // 流水线执行只能有一个并发
)
```

通过限制工作协程数为1和最大并发任务数为1，确保任务严格按照顺序执行。

## 3. 应用场景

流水线模式特别适合以下业务场景：

### 3.1 数据处理流水线

- **ETL 处理流程**：数据提取（Extract）、转换（Transform）、加载（Load）的多阶段处理
- **数据清洗与转换**：需要按顺序执行的数据预处理操作
- **批量数据处理**：对数据集合进行顺序处理，后续处理依赖前面的结果

### 3.2 报告生成系统

- **多步骤报告生成**：报告生成的各个阶段需要按顺序执行
- **报表汇总流程**：先生成子报表，再汇总生成总报表
- **数据分析流水线**：数据预处理、分析、可视化的顺序执行

### 3.3 事务性操作序列

- **跨系统操作序列**：需要在多个系统间按顺序执行的操作
- **状态转换流程**：对象状态变更需要严格按照指定顺序执行
- **资源初始化序列**：资源的创建和初始化需要按顺序进行

### 3.4 其他适合场景

- 任何需要保证任务执行顺序的业务流程
- 后续任务依赖于前面任务执行结果的场景
- 需要避免并发访问共享资源的场景

## 4. 使用方法

### 4.1 初始化混合调度器并启用流水线模式

要使用流水线模式，首先需要初始化混合调度器并配置流水线工作协程数：

```go
import (
    "github.com/DotNetAge/sparrow/pkg/tasks"
)

// 创建混合调度器配置
scheduler := tasks.NewHybridTaskScheduler(
    tasks.WithHybridWorkerCount(10, 2, 3),  // 并发10个worker，顺序2个，流水线3个
    tasks.WithHybridMaxConcurrentTasks(50), // 最大并发任务数
)
```

> 注意：流水线工作协程数（第三个参数）必须大于0才能启用流水线功能

### 4.2 注册流水线任务策略

为特定的任务类型注册流水线执行策略：

```go
// 注册流水线任务策略
scheduler.RegisterTaskPolicy("data-pipeline", tasks.PolicyPipeline)
scheduler.RegisterTaskPolicy("etl-process", tasks.PolicyPipeline)
scheduler.RegisterTaskPolicy("report-generation", tasks.PolicyPipeline)
```

### 4.3 提交流水线任务

使用任务构建器创建并提交流水线任务：

```go
// 创建并提交流水线任务
task := tasks.NewTaskBuilder().
    WithType("etl-process"). // 指定任务类型为已注册的流水线策略
    Immediate().             // 立即执行
    WithHandler(func(ctx context.Context) error {
        // 任务处理逻辑
        // 例如：执行ETL操作
        return processData(ctx)
    }).
    Build()

// 提交任务
err := scheduler.Schedule(task)
if err != nil {
    // 处理错误
}
```

## 5. 高级用法

### 5.1 使用 bootstrap 包简化配置

在 Sparrow 框架中，可以通过 bootstrap 包简化流水线任务的配置：

```go
import (
    "github.com/DotNetAge/sparrow/pkg/bootstrap"
)

// 创建应用实例
app := bootstrap.NewApp(
    bootstrap.HybridTasks(       // 使用混合任务调度器
        bootstrap.WithPipelineWorkers(3), // 设置流水线工作协程数
    ),
)

// 获取混合调度器实例
if hybridScheduler, ok := app.Scheduler.(*tasks.HybridTaskScheduler); ok {
    // 注册流水线任务策略
    hybridScheduler.RegisterTaskPolicy("data-pipeline", tasks.PolicyPipeline)
}

// 使用应用实例提交任务
taskId := app.RunTypedTask("data-pipeline", func(ctx context.Context) error {
    // 任务处理逻辑
    return nil
})
```

### 5.2 定时执行流水线任务

可以将流水线任务配置为定时执行：

```go
import "time"

// 创建定时执行的流水线任务
executeTime := time.Now().Add(1 * time.Hour)
task := tasks.NewTaskBuilder().
    WithType("report-generation").
    ScheduleAt(executeTime).     // 在指定时间执行
    WithHandler(func(ctx context.Context) error {
        // 生成报告的逻辑
        return generateDailyReport(ctx)
    }).
    Build()

scheduler.Schedule(task)
```

### 5.3 周期性执行流水线任务

也可以配置流水线任务周期性执行：

```go
// 创建周期性执行的流水线任务
task := tasks.NewTaskBuilder().
    WithType("data-pipeline").
    ScheduleRecurring(24 * time.Hour). // 每24小时执行一次
    WithHandler(func(ctx context.Context) error {
        // 周期性数据处理逻辑
        return processDailyData(ctx)
    }).
    Build()

scheduler.Schedule(task)
```

## 6. 最佳实践

### 6.1 任务类型设计

- **合理划分任务类型**：根据业务逻辑将任务分为不同类型，合理选择执行策略
- **流水线任务命名**：使用明确的命名方式，如 `{业务}-pipeline` 或 `{功能}-process`

### 6.2 性能优化

- **合理设置流水线工作协程数**：根据系统资源和任务量调整流水线工作协程数
- **任务粒度控制**：将大型流水线任务拆分为多个独立的小任务，以便更好地监控和管理
- **避免长时间阻塞**：流水线任务应避免长时间阻塞，影响后续任务执行

### 6.3 错误处理

- **设置错误处理回调**：为流水线任务设置错误处理回调函数

```go
task := tasks.NewTaskBuilder().
    WithType("data-pipeline").
    Immediate().
    WithHandler(func(ctx context.Context) error {
        // 任务处理逻辑
        return processData(ctx)
    }).
    WithOnComplete(func(ctx context.Context, err error) {
        if err != nil {
            // 处理错误，如记录日志、发送告警等
            log.Printf("流水线任务失败: %v", err)
        }
    }).
    Build()
```

### 6.4 与重试机制结合

流水线任务可以与系统的重试机制结合使用：

```go
// 启用重试机制
app := bootstrap.NewApp(
    bootstrap.HybridTasks(),
    bootstrap.WithRetry(        // 启用重试配置
        bootstrap.WithMaxRetries(3),
        bootstrap.WithExponentialBackoff(2.0),
    ),
)

// 提交流水线任务，自动支持重试
app.RunTypedTask("data-pipeline", func(ctx context.Context) error {
    // 任务处理逻辑
    return processData(ctx)
})
```

## 7. 监控与统计

### 7.1 获取流水线任务统计信息

可以通过以下方式获取流水线任务的执行统计信息：

```go
// 获取混合调度器统计信息
stats := scheduler.GetStats()

// 获取流水线任务统计
pipelineStats := stats[tasks.PolicyPipeline]
fmt.Printf("流水线任务统计: 总计=%d, 已完成=%d, 失败=%d, 运行中=%d\n",
    pipelineStats.TotalTasks,
    pipelineStats.CompletedTasks,
    pipelineStats.FailedTasks,
    pipelineStats.RunningTasks,
)
```

### 7.2 监控任务执行状态

定期监控流水线任务的执行状态，确保任务按预期执行：

```go
func monitorPipelineStatus(scheduler *tasks.HybridTaskScheduler) {
    ticker := time.NewTicker(1 * time.Minute)
    defer ticker.Stop()
    
    for range ticker.C {
        stats := scheduler.GetStats()
        if pipelineStats, exists := stats[tasks.PolicyPipeline]; exists {
            log.Printf("流水线监控: 总任务=%d, 运行中=%d, 最后执行=%v\n",
                pipelineStats.TotalTasks,
                pipelineStats.RunningTasks,
                pipelineStats.LastExecution,
            )
        }
    }
}
```

## 8. 与其他执行策略的比较

| 执行策略 | 工作协程数 | 最大并发 | 执行顺序 | 适用场景 |
|---------|-----------|---------|---------|----------|
| 并发模式 (Concurrent) | 多个 | 可配置 | 无序 | 独立、无依赖的任务 |
| 顺序模式 (Sequential) | 1个 | 1个 | 严格顺序 | 对共享资源有并发访问限制的任务 |
| 流水线模式 (Pipeline) | 1个 | 1个 | 严格顺序 | 需要按顺序执行的多阶段处理任务 |

## 9. 流水线模式与顺序模式的区别

虽然流水线模式和顺序模式在工作协程数、最大并发数和执行顺序上看起来相似，但它们在内部实现、设计目的和适用场景上存在重要区别。

### 9.1 设计目标与本质区别

**顺序模式**
- 设计目标：提供一个简单的单线程执行环境，避免并发问题
- 本质特性：是并发模式的一个特例，主要用于隔离可能存在并发安全问题的任务
- 实现方式：在混合调度器中，顺序模式是通过一个工作协程的并发调度器实现的

**流水线模式**
- 设计目标：提供一个专门的执行环境，用于处理具有前后依赖关系的任务序列
- 本质特性：是一个独立的执行通道，专注于保证任务的严格顺序性
- 实现方式：拥有独立的调度器实例，完全与其他调度策略隔离

### 9.2 内部实现机制差异

```go
// 顺序模式实现（简化）
func (s *HybridTaskScheduler) handleSequentialTask(task *Task) {
    // 在同一个工作协程中执行所有顺序任务
    // 不区分任务类型，所有标记为顺序的任务都在同一个通道中排队
}

// 流水线模式实现（简化）
func (s *HybridTaskScheduler) handlePipelineTask(task *Task) {
    // 流水线模式拥有独立的调度器实例
    // 每个流水线任务都通过独立的机制提交和执行
}
```

### 9.3 关键功能区别

1. **任务隔离度**
   - 顺序模式：所有顺序任务共享同一个工作协程，不同类型的顺序任务会互相影响
   - 流水线模式：通过配置多个流水线工作协程，可以将不同类型的流水线任务隔离在不同的执行通道中

2. **资源分配灵活性**
   - 顺序模式：资源分配固定，无法针对不同类型的顺序任务进行差异化配置
   - 流水线模式：可以通过配置不同数量的流水线工作协程，为不同类型的流水线任务分配不同的资源

3. **状态管理**
   - 顺序模式：通常用于独立任务，不强调任务之间的状态传递
   - 流水线模式：更适合处理需要在任务之间保持某种上下文或状态的场景

4. **失败隔离**
   - 顺序模式：一个顺序任务失败可能会影响其他不相关的顺序任务
   - 流水线模式：可以通过多个流水线工作协程将不同业务的流水线任务隔离，减少故障影响范围

### 9.4 使用场景的明确区分

**顺序模式适用场景**：
- 需要避免并发问题但相互之间无直接关联的独立任务
- 简单的单线程执行环境需求
- 对共享资源的访问隔离需求
- 不需要精细资源控制的场景

**流水线模式适用场景**：
- 有明确前后依赖关系的任务序列
- 业务流程中需要保证严格执行顺序的多个步骤
- 需要在不同任务之间传递状态或上下文的场景
- 需要针对不同业务流程进行资源隔离和精细化控制的场景
- ETL处理、数据转换链、多步骤报告生成等复杂业务流程

### 9.5 性能与资源利用特点

- **顺序模式**：更轻量级，但缺乏隔离性，任务之间可能相互阻塞
- **流水线模式**：提供更好的隔离性和可控性，但需要更多的资源配置

### 9.6 实际应用案例比较

**顺序模式案例**：
- 定期清理临时文件的独立任务
- 需要串行执行但彼此不相关的日志分析任务
- 简单的数据同步操作

**流水线模式案例**：
- 完整的ETL流程（提取、转换、加载）
- 多阶段的数据处理流水线
- 复杂的报告生成系统（数据收集、分析、汇总、生成）
- 需要保持业务流程完整性的跨系统操作
- **订单处理流程**：订单创建、库存检查、支付处理、物流安排等一系列有依赖关系的步骤

## 10. 常见问题与解决方案

## 11. 常见问题与解决方案

### 11.1 流水线任务阻塞

**问题**：流水线中的某个任务执行时间过长，阻塞了后续任务。

**解决方案**：
- 优化任务处理逻辑，减少执行时间
- 将大型任务拆分为多个小型任务
- 考虑设置任务超时机制

### 11.2 流水线任务失败导致阻塞

**问题**：流水线任务失败后，由于重试机制，可能长时间占用流水线。

**解决方案**：
- 设置合理的重试次数和退避策略
- 在失败处理回调中及时释放资源
- 考虑使用死信队列处理持续失败的任务

### 11.3 流水线工作协程数配置不当

**问题**：流水线工作协程数过多或过少。

**解决方案**：
- 过少：增加流水线工作协程数，将不同类型的流水线任务分配到不同的协程
- 过多：减少流水线工作协程数，避免资源浪费
- 根据系统负载和任务量动态调整

## 12. 输入输出示例

### 示例1：基本的流水线任务

#### 输入：
```go
import (
    "context"
    "fmt"
    "github.com/DotNetAge/sparrow/pkg/tasks"
)

func main() {
    // 创建混合调度器
    scheduler := tasks.NewHybridTaskScheduler(
        tasks.WithHybridWorkerCount(5, 0, 1), // 启用1个流水线worker
    )
    
    // 注册流水线策略
    scheduler.RegisterTaskPolicy("order-process", tasks.PolicyPipeline)
    
    // 启动调度器
    scheduler.Start(context.Background())
    
    // 提交3个流水线任务
    for i := 1; i <= 3; i++ {
        taskID := fmt.Sprintf("order-%d", i)
        task := tasks.NewTaskBuilder().
            WithID(taskID).
            WithType("order-process").
            Immediate().
            WithHandler(func(ctx context.Context) error {
                fmt.Printf("开始处理任务: %s\n", taskID)
                // 模拟处理时间
                time.Sleep(1 * time.Second)
                fmt.Printf("完成任务: %s\n", taskID)
                return nil
            }).
            Build()
        
        scheduler.Schedule(task)
    }
    
    // 等待任务完成
    time.Sleep(5 * time.Second)
    scheduler.Close(context.Background())
}
```

#### 输出：
```
开始处理任务: order-1
完成任务: order-1
开始处理任务: order-2
完成任务: order-2
开始处理任务: order-3
完成任务: order-3
```

    "errors"
    "fmt"
    "github.com/DotNetAge/sparrow/pkg/tasks"

        tasks.WithHybridWorkerCount(5, 0, 2), // 2个流水线工作协程

    scheduler.Start(context.Background())
    defer scheduler.Close(context.Background())
    
    // 注册订单处理流水线策略
    scheduler.RegisterTaskPolicy("order-processing", tasks.PolicyPipeline)
    
    // 订单数据结构
    type OrderContext struct {
        OrderID    string
        ProductID  string
        Quantity   int
        UserID     string
        Paid       bool
        InventoryReserved bool
    }
    
    // 创建订单上下文
    orderCtx := &OrderContext{
        OrderID:   "ORD-" + time.Now().Format("20060102150405"),
        ProductID: "PROD-123",
        Quantity:  2,
        UserID:    "USER-456",
    }
    
    // 1. 创建订单任务
    createOrderTask := tasks.NewTaskBuilder().
        WithID(orderCtx.OrderID + "-create").
        WithType("order-processing").
        Immediate().
        WithHandler(func(ctx context.Context) error {
            fmt.Printf("[订单:%s] 创建订单记录\n", orderCtx.OrderID)
            // 模拟创建订单数据库操作
            return nil
        }).
        Build()
    
    // 2. 库存检查与预留任务
    reserveInventoryTask := tasks.NewTaskBuilder().
        WithID(orderCtx.OrderID + "-inventory").
        WithType("order-processing").
        Immediate().
        WithRetry(func(t *tasks.RetryOptions) {
            t.MaxRetries = 3         // 最多重试3次
            t.Interval = 2 * time.Second // 初始间隔2秒
            t.BackoffMultiplier = 1.5    // 退避因子
        }).
        WithHandler(func(ctx context.Context) error {
            fmt.Printf("[订单:%s] 检查并预留库存: %s x %d\n",

            // 此处可以添加随机失败来测试重试机制
            orderCtx.InventoryReserved = true
            return nil
        }).
        WithOnComplete(func(ctx context.Context, err error) {
            if err != nil {
                fmt.Printf("[订单:%s] 库存预留失败: %v，触发补偿操作\n", orderCtx.OrderID, err)
                // 触发补偿操作，如取消订单
                // 这里可以实现Saga模式的补偿事务
            }
        }).
        Build()
    
    // 3. 支付处理任务
    paymentTask := tasks.NewTaskBuilder().
        WithID(orderCtx.OrderID + "-payment").
        WithType("order-processing").
        Immediate().
        WithRetry(func(t *tasks.RetryOptions) {
            t.MaxRetries = 3
            t.Interval = 5 * time.Second // 支付操作重试间隔更长
        }).
        WithHandler(func(ctx context.Context) error {
            if !orderCtx.InventoryReserved {
                return errors.New("库存未预留，无法处理支付")
            }
            fmt.Printf("[订单:%s] 处理支付，用户: %s\n", orderCtx.OrderID, orderCtx.UserID)
            // 模拟支付处理
            orderCtx.Paid = true
            return nil
        }).
        WithOnComplete(func(ctx context.Context, err error) {
            if err != nil {
                fmt.Printf("[订单:%s] 支付失败: %v，触发补偿操作\n", orderCtx.OrderID, err)
                // 补偿操作：释放预留的库存
            }
        }).
        Build()
    
    // 4. 物流安排任务
    shippingTask := tasks.NewTaskBuilder().
        WithID(orderCtx.OrderID + "-shipping").
        WithType("order-processing").
        Immediate().
        WithHandler(func(ctx context.Context) error {
            if !orderCtx.Paid {
                return errors.New("订单未支付，无法安排物流")
            }
            fmt.Printf("[订单:%s] 安排物流配送\n", orderCtx.OrderID)
            // 模拟物流安排
            return nil
        }).
        Build()
    
    // 按顺序提交订单处理流水线任务
    fmt.Println("开始订单处理流水线...")
    scheduler.Schedule(createOrderTask)
    scheduler.Schedule(reserveInventoryTask)
    scheduler.Schedule(paymentTask)
    scheduler.Schedule(shippingTask)
    
    // 等待任务完成
    time.Sleep(20 * time.Second)
    fmt.Println("订单处理流水线完成")
}

### 流水线模式实现简单事务或Saga的说明

通过上述订单处理示例，我们可以看到流水线模式如何结合重试规则实现简单的事务处理或Saga形式：

1. **严格的执行顺序保证**：订单创建、库存预留、支付处理、物流安排等步骤严格按照顺序执行

2. **任务依赖管理**：每个后续任务可以检查前序任务的执行结果，如支付任务检查库存是否已预留

3. **错误处理与重试机制**：对关键步骤（如库存预留和支付）配置重试策略，提高系统可靠性

4. **补偿事务支持**：通过`WithOnComplete`回调函数实现失败时的补偿操作，符合Saga模式的设计理念
   - 库存预留失败时取消订单
   - 支付失败时释放已预留的库存

5. **业务流程完整性**：整个订单处理流程作为一个完整的业务事务进行管理

这种模式特别适合分布式系统中的长事务处理，可以避免传统ACID事务的局限性，同时保持业务流程的一致性。流水线模式提供的严格顺序执行保证，为实现可靠的业务流程提供了坚实的基础。

通过以上示例可以看出，流水线模式确保任务按照提交顺序严格执行，非常适合需要按顺序执行的多阶段处理流程，特别是结合重试规则和补偿机制后，可以实现强大的业务事务处理能力。
