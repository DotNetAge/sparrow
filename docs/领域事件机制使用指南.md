# 领域事件机制使用指南

领域事件通信与事件总线虽然字面上同为“事件”的传递含义，但在具体应用上却存在差异。
首先，领域事件通信是事件总线的一种应用机制。事件总线只负责服务自身或多个分布式服务之间的一通用事件传递，它们都使用同一种事件格式，事件具体数据是一种通用的数据负载。而的领域事件则是一种面向领域模型，分离技术关注点的事件抽象。领域事件通信只关注事件的服务方，从属的聚合类型和具体的事件类别。这就是两者最为之具体的区别。

因此两者在使用上有各自的类去负责处理。

再次强调：**领域事件通信与事件通信是形式相似而本质不同的两种事件通信机制，事件总线则重于消息的传递偏向于技术，领域事件通信则重于界定服务边界，向服务边界发布上下文内容，偏向于业务**

# 领域事件模块 (messaging)

本模块提供了领域事件的发布和订阅机制，连接事件存储与事件总线，是实现事件驱动架构的关键组件。

## 目录结构

- `README.md` - 模块使用说明
- `publisher.go` - 实现事件发布器
- `subscriber.go` - 实现事件订阅器

## 核心组件说明

### EventPublisher 组件

事件发布器负责发布领域事件，封装了`EventBus`和`EventStore`的操作。它实现了"服务名.聚合类.事件名"的主题命名规则，确保事件能够被正确路由到相应的订阅者。

### EventSubscriber 组件

事件订阅器负责订阅事件总线上的领域事件，根据不同事件类型、聚合类型和服务名称将其分发给对应的处理器。它同时负责对事件总线的事件类型进行主题(Topic)编码，格式为：服务名.聚合类型.事件类型。

**关键特性：**

- 基于泛型实现，每个订阅器实例专门用于订阅和处理某一特定类型的领域事件
- 支持根据服务名称和聚合类型过滤事件
- 支持聚合类型使用通配符`*`订阅所有类型的聚合事件
- 自动处理通用事件到特定领域事件的转换

### DomainEventHandler 函数类型

领域事件处理函数，专门用于处理特定类型的领域事件。

```go
// T 为领域事件类型
type DomainEventHandler[T entity.DomainEvent] func(ctx context.Context, event T) error
```

## EventPublisher 组件使用指南

### 初始化发布器

```go
import "github.com/DotNetAge/sparrow/pkg/messaging"

// eventStore 是实现了 usecase.EventStore 接口的实例
// eventBus 是实现了 eventbus.EventBus 接口的实例
// serviceName 是当前服务的名称，用于构建事件主题
eventPublisher := messaging.NewEventPublisher(eventStore, eventBus, "order-service")
```

### 发布聚合的所有事件

```go
// 通过聚合根ID获取并发布所有事件
err := eventPublisher.PublishEvents(ctx, "aggregate-id")
if err != nil {
    // 处理错误
}
```

### 发布聚合根中未提交的事件

```go
// aggregate 是实现了 entity.AggregateRoot 接口的聚合根实例
err := eventPublisher.PublishUncommittedEvents(ctx, aggregate)
if err != nil {
    // 处理错误
}
```

### 内部工作机制

EventPublisher会自动将领域事件转换为通用事件，并按照"服务名.聚合类.事件名"的格式构建完整的事件主题，确保事件能够被正确路由到相应的订阅者。事件发布过程中会将领域事件的完整信息包含在通用事件的Payload中。

## EventSubscriber 组件使用指南

EventSubscriber组件基于Go泛型实现，每个订阅器实例专门用于订阅和处理某一特定类型的领域事件。

### 初始化订阅器

```go
import (
    "github.com/DotNetAge/sparrow/pkg/messaging"
    "github.com/DotNetAge/sparrow/pkg/entity"
    // 引入具体的领域事件类型
    "your-project/pkg/domain/events"
)

// eventBus 是实现了 eventbus.EventBus 接口的实例
// serviceName 是当前服务名称
// aggregateType 是聚合类型，设置为"*"可以接收所有聚合类型的事件
// logger 是日志记录器实例

// 创建一个订阅器实例，用于订阅特定类型的领域事件
var subscriber messaging.EventSubscriber[events.OrderCreatedEvent]
subscriber.Init("payment-service", "order", eventBus, logger)

// 或者创建订阅所有聚合类型的订阅器
var wildcardSubscriber messaging.EventSubscriber[events.OrderCreatedEvent]
wildcardSubscriber.Init("payment-service", "*", eventBus, logger)
```

### 添加事件处理器

```go
// 添加处理器，接收当前服务的所有订单创建事件
err := subscriber.AddHandler(func(ctx context.Context, event events.OrderCreatedEvent) error {
    // 处理订单创建事件
    // 例如：创建支付记录
    return nil
})
if err != nil {
    // 处理错误
}

// 添加处理器，只接收来自特定服务的事件
err = subscriber.AddServiceHandler("order-service", func(ctx context.Context, event events.OrderCreatedEvent) error {
    // 处理来自order-service的订单创建事件
    return nil
})
if err != nil {
    // 处理错误
}
```

### 取消订阅

```go
// 取消订阅当前服务的事件
err := subscriber.Unsubscribe("payment-service")
if err != nil {
    // 处理错误
}

// 取消订阅特定服务的事件
err = subscriber.Unsubscribe("order-service")
if err != nil {
    // 处理错误
}
```

### 内部工作机制

1. **主题编码**：EventSubscriber使用`encodeEventType`方法将事件类型编码为符合"服务名.聚合类型.事件类型"格式的主题名称。
2. **事件过滤**：在处理事件时，会根据服务名和聚合类型进行过滤，确保只有匹配的事件被传递给处理器。
3. **通配符支持**：当聚合类型设置为`*`时，会接收所有类型的聚合事件。
4. **事件转换**：自动从通用事件中提取并转换为特定类型的领域事件，然后传递给注册的处理器。

## 跨消息服务框架兼容性

本模块设计考虑了与不同消息服务框架（如NATS、RabbitMQ、Redis等）的兼容性：

- 统一使用`.`作为主题分隔符，确保在不同消息服务框架间的一致性
- 遵循各消息服务框架的通配符规范，如Redis支持`*`、`?`等通配符
- 事件总线实现负责处理与具体消息服务框架的交互细节

## 最佳实践

1. **每个事件类型一个订阅器**：为每种领域事件类型创建单独的订阅器实例，保持代码清晰和类型安全
2. **合理使用通配符**：在需要接收多种聚合类型事件时使用`*`通配符，但注意可能带来的性能影响
3. **服务名称一致性**：在发布和订阅事件时使用一致的服务名称，确保事件能够正确路由
4. **错误处理**：妥善处理事件订阅和处理过程中的错误，避免影响系统稳定性
5. **日志记录**：利用logger记录事件订阅、取消订阅和处理过程中的关键信息，便于调试和监控

## 示例场景

### 订单支付处理流程

1. 订单服务发布`OrderCreatedEvent`事件，主题为`order-service.order.OrderCreatedEvent`
2. 支付服务创建`EventSubscriber[OrderCreatedEvent]`订阅器，初始化时指定服务名`payment-service`和聚合类型`order`
3. 支付服务添加处理器，处理接收到的订单创建事件并创建相应的支付记录
4. 当订单服务发布事件后，支付服务的订阅器会自动接收并处理匹配的事件

这种基于事件的异步通信模式，使得系统各组件松耦合，提高了系统的可扩展性和可靠性。
