# 事件总线使用指南

## 1. 事件总线概述

事件总线是一个实现发布/订阅模式的通信机制，用于组件之间的解耦通信。在 Sparrow 框架中，事件总线提供了一个统一的接口，使系统中的不同部分能够通过发布和订阅事件进行交互，而不需要直接相互引用。

### 1.1 主要特点

- **统一接口**：所有事件总线实现都遵循相同的接口定义
- **多种实现**：支持内存、Redis和NATS JetStream等多种实现方式
- **异步通信**：支持异步事件处理，提高系统响应性
- **类型化事件**：通过事件类型进行事件的分类和路由
- **线程安全**：所有实现都保证线程安全，支持并发操作

## 2. 接口定义

事件总线接口定义在 `pkg/eventbus/eventbus.go` 文件中，包含以下核心方法：

```go
// EventBus 定义事件总线接口
// 所有事件总线实现必须满足这个接口
// 支持发布/订阅模式
type EventBus interface {
    // Pub 发布事件到事件总线
    // ctx: 上下文，用于超时控制和取消操作
    // evt: 要发布的事件
    // 返回值: 错误信息
    Pub(ctx context.Context, evt entity.Event) error

    // Sub 订阅指定类型的事件
    // eventType: 事件类型
    // handler: 事件处理器函数
    // 返回值: 错误信息
    Sub(eventType string, handler EventHandler) error

    // Unsub 取消订阅指定类型的事件
    // eventType: 事件类型
    // 返回值: 错误信息
    Unsub(eventType string) error

    // Close 关闭事件总线，释放资源
    // 返回值: 错误信息
    Close() error
}

// EventHandler 事件处理器接口
type EventHandler func(ctx context.Context, event entity.Event) error
```

### 2.1 事件接口

所有通过事件总线传递的事件都必须实现 `entity.Event` 接口：

```go
// Event 通用事件接口，是所有事件的基础接口
type Event interface {
    GetEventID() string      // 事件唯一标识符
    GetEventType() string    // 事件类型
    GetCreatedAt() time.Time // 事件创建时间
}
```

## 3. 事件总线实现

Sparrow 框架提供了五种事件总线实现，适用于不同的场景：

### 3.1 内存事件总线 (MemoryEventBus)

基于内存的轻量级事件总线实现，适用于单实例应用或测试环境。

**主要特点**：
- 纯 Go 实现，无需外部依赖
- 最高性能，无网络开销
- 不支持跨实例通信
- 应用重启后事件会丢失

**适用场景**：
- 单实例应用内部组件通信
- 单元测试和集成测试
- 开发环境快速原型开发

### 3.2 Redis 事件总线 (RedisBus)

基于 Redis 发布/订阅功能的事件总线实现，支持多实例间的事件通信。

**主要特点**：
- 支持多实例、分布式环境
- 基于 Redis 的发布/订阅机制
- 轻量级，易于部署和配置
- 不支持事件持久化（默认配置下）

**适用场景**：
- 分布式系统的组件通信
- 需要跨实例同步状态的场景
- 对消息可靠性要求适中的应用

### 3.3 RabbitMQ 事件总线 (RabbitMQEventBus)

基于 RabbitMQ 消息队列的事件总线实现，支持可靠的消息传递和灵活的路由机制。

**主要特点**：
- 支持消息确认机制
- 提供灵活的路由模式（通过交换机和队列）
- 支持消息持久化
- 高可用性和容错能力

**适用场景**：
- 需要可靠消息传递的分布式系统
- 复杂的事件路由需求
- 对消息顺序和优先级有要求的应用

### 3.4 NATS JetStream 事件总线 (JetStreamBus)

基于 NATS JetStream 的高级事件总线实现，提供持久化和可靠的消息传递。

**主要特点**：
- 支持消息持久化和重播
- 提供消息确认机制
- 支持消息流控制和背压
- 高可用性和容错能力

**适用场景**：
- 对消息可靠性要求高的场景
- 需要消息持久化和重播的应用
- 复杂的分布式系统架构
- 关键业务流程中的事件传递

### 3.5 Kafka 事件总线 (KafkaEventBus)

基于 Apache Kafka 的高性能分布式事件总线实现，提供高吞吐量、持久化和可靠的事件流处理。

**主要特点**：
- 高吞吐量，适合大规模事件处理
- 消息持久化存储，支持长期数据保留
- 分布式架构，支持高可用性
- 支持精确一次处理语义
- 分区和复制机制确保数据可靠性

**适用场景**：
- 大规模分布式系统的事件处理
- 需要高吞吐量的实时数据管道
- 事件溯源和事件驱动架构
- 数据集成和流处理场景

## 4. 使用方法

### 4.1 创建事件总线实例

#### 内存事件总线

```go
import "github.com/DotNetAge/sparrow/pkg/eventbus"

// 创建内存事件总线实例
bus := eventbus.NewMemoryEventBus()
```

#### Redis 事件总线

```go
import (
    "github.com/DotNetAge/sparrow/pkg/config"
    "github.com/DotNetAge/sparrow/pkg/eventbus/redis"
)

// 创建Redis配置
redisConfig := &config.RedisConfig{
    Host:     "localhost",
    Port:     6379,
    Password: "", // 如果Redis没有设置密码则为空
    DB:       0,
}

// 创建Redis事件总线实例
bus, err := redis.NewRedisEventBus(redisConfig)
if err != nil {
    // 处理错误
    log.Fatalf("创建Redis事件总线失败: %v", err)
}
```

#### RabbitMQ 事件总线

```go
import (
    "github.com/DotNetAge/sparrow/pkg/config"
    "github.com/DotNetAge/sparrow/pkg/eventbus/rabbitmq"
)

// 创建RabbitMQ配置
rabbitConfig := &config.RabbitMQConfig{
    Host:     "localhost",
    Port:     5672,
    Username: "guest",
    Password: "guest",
    VHost:    "/",
    Exchange: "events",
}

// 创建RabbitMQ事件总线实例
bus, err := rabbitmq.NewRabbitMQEventBus(rabbitConfig)
if err != nil {
    // 处理错误
    log.Fatalf("创建RabbitMQ事件总线失败: %v", err)
}
```

#### NATS JetStream 事件总线

```go
import (
    "github.com/DotNetAge/sparrow/pkg/config"
    "github.com/DotNetAge/sparrow/pkg/eventbus/jetstream"
)

// 创建NATS配置

    NATSURL:    "nats://localhost:4222",
    StreamName: "events",
    DurableName: "my-consumer", // 可选，用于持久化订阅
}

// 创建NATS JetStream事件总线实例

if err != nil {
    // 处理错误

```

#### Kafka 事件总线

```go

    "github.com/DotNetAge/sparrow/pkg/config"
    "github.com/DotNetAge/sparrow/pkg/eventbus/kafka"

    GroupID: "my-consumer-group",       // 消费者组ID

// 创建Kafka事件总线实例

    // 处理错误

}
```

### 4.2 发布事件

```go
import (
    "context"
    "time"
    "github.com/DotNetAge/sparrow/pkg/entity"
)

// 定义一个自定义事件
type UserCreatedEvent struct {
    ID        string    `json:"id"`
    Username  string    `json:"username"`
    Email     string    `json:"email"`
    CreatedAt time.Time `json:"created_at"`
}

// 实现Event接口
func (e *UserCreatedEvent) GetEventID() string {
    return e.ID
}

func (e *UserCreatedEvent) GetEventType() string {
    return "user_created"
}

func (e *UserCreatedEvent) GetCreatedAt() time.Time {
    return e.CreatedAt
}

// 创建并发布事件
ctx := context.Background()
event := &UserCreatedEvent{
    ID:        "user-123",
    Username:  "johndoe",
    Email:     "john@example.com",
    CreatedAt: time.Now(),
}

if err := bus.Pub(ctx, event); err != nil {
    // 处理发布错误
    log.Printf("发布事件失败: %v", err)
}
```

### 4.3 订阅事件

```go
import "context"

// 订阅事件类型
err := bus.Sub("user_created", func(ctx context.Context, event entity.Event) error {
    // 将事件转换为具体类型
    userEvent, ok := event.(*UserCreatedEvent)
    if !ok {
        return fmt.Errorf("无法转换为UserCreatedEvent类型")
    }
    
    // 处理事件
    fmt.Printf("收到用户创建事件: %s, 用户名: %s\n", userEvent.ID, userEvent.Username)
    
    // 执行业务逻辑...
    
    return nil
})

if err != nil {
    // 处理订阅错误
    log.Printf("订阅事件失败: %v", err)
}
```


## 5. 各实现之间的比较

| **依赖**       | 无外部依赖       | Redis服务器                | RabbitMQ服务器             | NATS服务器                 | Kafka集群                  |

| **可靠性**     | 低（进程内）     | 中（无持久化）             | 高（持久化和确认）         | 高（持久化和确认）         | 极高（复制和持久化）       |
| **分布式支持** | 不支持           | 支持                       | 支持                       | 支持                       | 支持（原生分布式）         |
| **持久化**     | 不支持           | 可选（需额外配置）         | 支持                       | 支持                       | 支持（长期存储）           |
| **消息重播**   | 不支持           | 不支持                     | 支持（通过队列重播）       | 支持                       | 支持（通过偏移量）         |
| **消息确认**   | 不支持           | 不支持                     | 支持                       | 支持                       | 支持（多种确认模式）       |

| **适用场景**   | 单实例应用、测试 | 分布式系统、一般可靠性要求 | 复杂路由、高可靠消息传递   | 高可靠性要求、关键业务流程 | 大规模数据处理、事件溯源   |
| **部署复杂度** | 非常简单         | 中等                       | 中等                       | 较高                       | 高（需要ZooKeeper或KRaft） |

## 6. 最佳实践

1. **选择合适的实现**：根据应用的规模、分布式需求和可靠性要求选择合适的事件总线实现。
2. **错误处理**：始终处理事件发布和订阅过程中可能出现的错误。
3. **上下文管理**：正确使用上下文进行超时控制和取消操作。
4. **资源释放**：在应用关闭时正确调用 Close 方法释放资源。
5. **事件设计**：
   - 使用有意义的事件类型名称
   - 事件数据应该是不可变的
   - 包含足够的上下文信息
   - 避免过大的事件负载
6. **处理器设计**：
   - 处理器应该快速执行，避免阻塞
   - 处理器内部的错误不应影响事件总线运行
   - 必要时实现重试机制
7. **监控和日志**：为事件处理过程添加适当的日志和监控，以便追踪事件流和排查问题。

## 7. 常见问题解答

### 7.1 如何实现事件的幂等性处理？

在分布式系统中，由于网络问题可能导致事件重复投递，建议：

- 为每个事件分配唯一标识符
- 在事件处理器中实现基于事件ID的去重逻辑
- 使用幂等操作确保多次处理同一个事件不会产生副作用

### 7.2 如何处理事件处理失败的情况？

根据应用需求，可以采用以下策略：

- **重试机制**：实现有限次数的重试逻辑
- **死信队列**：将处理失败的事件发送到专门的队列进行后续处理
- **错误日志和告警**：记录失败事件并触发告警以便及时处理

### 7.3 如何在微服务架构中使用事件总线？

- 选择支持分布式的实现（Redis、RabbitMQ或NATS）
- 定义清晰的事件契约和版本管理策略
- 实现服务间的事件一致性确保机制
- 使用消息验证和授权机制保护事件流

### 7.4 如何进行性能优化？

- 避免在事件处理器中执行耗时操作
- 合理设置批量处理和并发处理参数
- 监控事件总线性能指标，及时发现瓶颈
- 根据负载情况进行适当的水平扩展

通过遵循本指南，您可以有效地使用 Sparrow 框架的事件总线功能，实现系统组件之间的松耦合通信，提高系统的可扩展性和可维护性。
