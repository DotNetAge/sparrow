引言​

本文件系统化阐述 Sparrow 框架的分层架构设计与实现，聚焦应用层、适配器层、领域层、基础设施层的职责边界、依赖方向与交互模式。通过依赖注入容器、仓储接口、事件总线与 HTTP 适配器等关键构件，实现关注点分离与松耦合高内聚。文档同时给出层间接口设计、依赖注入使用、错误传播机制与最佳实践建议，帮助开发者快速理解并正确扩展框架。

项目结构​

Sparrow 采用“整洁架构”风格，围绕用例层组织业务逻辑，领域层承载实体与聚合根，基础设施层提供持久化、消息与运行时环境，适配器层负责外部协议接入（如 HTTP）。核心模块分布如下：

  • 应用层（用例层）：usecase/* 提供服务编排与业务规则，如会话服务、仓储接口与基础实现。
  • 领域层：entity/* 定义实体与聚合根接口，承载业务不变量。
  • 适配器层：adapter/http/* 提供 HTTP 请求处理与路由注册。
  • 基础设施层：bootstrap/（容器、配置、数据库）、persistence/repo/（多种存储实现）、messaging/*（事件总线与流处理）。
  • 应用层（用例层）
    • 会话服务：封装会话生命周期与数据读写，依赖仓储接口。
    • 仓储接口与基础实现：统一 CRUD 与查询能力，支持分页、条件查询、随机取样等。
  • 领域层
    • 聚合根接口：统一事件溯源、版本控制与业务验证。
    • 任务实体：承载任务状态机与重试策略。
  • 适配器层
    • HTTP 处理器：将 HTTP 请求映射到用例层服务；路由注册由适配器层负责。
  • 基础设施层
    • 应用程序主容器：全局配置、日志、路由引擎、服务容器与子进程管理。
    • 依赖注入容器：支持匿名与具名依赖注册与解析。
    • 数据库抽象：统一 Redis、SQL、Badger 等客户端。
    • 事件总线与流处理：基于 NATS JetStream 的 Hub/Bus 抽象。

Sparrow 的分层边界与交互如下：

  • 依赖方向：上层依赖下层接口，下层不依赖上层。
  • 关注点分离：适配器层仅负责协议转换；用例层专注业务编排；领域层维护不变量；基础设施层提供通用能力。
  • 松耦合高内聚：通过接口与依赖注入实现解耦；每层内部围绕单一职责组织。

应用层（用例层）​

  • 会话服务
    • 职责：创建、获取、更新、删除会话；维护过期时间；清理过期会话。
    • 依赖：仓储接口（Repository[*entity.Session]），日志与 TTL 配置。
    • 错误传播：内部捕获并记录错误，向上返回统一错误类型。
  • 仓储接口与基础实现
    • 接口：统一 CRUD、批量操作、分页、条件查询、随机取样等。
    • 基类：提供空实现与通用校验（如 ID 非空、分页参数默认值）。
  • 聚合根接口
    • 职责：统一聚合标识、版本控制、事件溯源（命令处理、事件应用、未提交事件管理、从事件/快照重建状态）与业务验证。
  • 任务实体
    • 职责：任务状态机（待执行/运行中/已完成/失败/取消）、重试控制、执行时长计算、过期判断。
  • HTTP 处理器
    • 会话处理器：将 HTTP 请求映射到会话服务，处理创建、获取、更新数据、删除与获取特定数据。
    • 任务处理器：预留接口，体现适配器层对任务用例的接入方式。
  • 路由注册：适配器层负责将处理器绑定到路由。
  • 应用程序主容器
    • 职责：加载配置、创建日志、初始化路由引擎、注册服务容器、管理子进程与优雅关闭。
    • 依赖注入：通过容器解析命名或匿名依赖，如会话服务、事件发布器、NATS 连接等。
  • 依赖注入容器
    • 能力：注册匿名/具名构造器、解析依赖链、缓存实例（单例行为）。
  • 数据库抽象
    • 统一 Redis、SQL、Badger 客户端，便于替换与扩展。
  • 事件总线与流处理
    • StreamHub：封装订阅者集合，支持启动与优雅关闭；基于 NATS JetStream 的 Hub/Bus。
  • 层间依赖方向
    • 适配器层 → 用例层：HTTP 处理器依赖会话服务。
    • 用例层 → 领域层：会话服务依赖仓储接口；仓储接口与基础实现不依赖上层。
    • 基础设施层 → 用例层：容器与数据库抽象为用例层提供运行时支撑。
  • 依赖注入与接口契约
    • 用例层通过接口编程，避免直接依赖具体实现；基础设施层通过容器注册具体实现。
    • 容器支持具名与匿名依赖，便于多实现切换与组合。
  • 仓储实现
    • SQL 仓储支持软删除、事务批处理、条件查询与排序分页，适合关系型场景；注意索引与查询条件优化。
  • 事件总线
    • 基于 NATS JetStream 的 Hub/Bus，支持多消费者与流式处理；合理设置流保留策略与消费者组。
  • 依赖注入
    • 单例缓存减少重复构造成本；避免循环依赖导致的解析失败。
  • HTTP 适配器
    • Gin 默认中间件与超时配置；注意路由层级与处理器职责单一，避免阻塞。

故障排查指南​

  • 依赖解析失败
    • 现象：解析具名/匿名依赖时报错。
    • 排查：确认构造器签名与返回值类型匹配；检查注册顺序与名称一致性。
  • 仓储操作异常
    • 现象：保存/更新/删除报错或返回未实现错误。
    • 排查：确认具体仓储实现已注册；检查实体 ID 与字段约束；核对数据库连接与权限。
  • 会话服务错误
    • 现象：创建/获取/更新/删除失败。
    • 排查：查看日志输出；确认 TTL 配置与过期清理逻辑；核对仓储实现。
  • 事件总线/流处理
    • 现象：订阅失败或无法启动。
    • 排查：检查 NATS 连接与 JetStream 流配置；确认 Hub/Bus 的启动与关闭流程。

Sparrow 通过清晰的分层边界与接口契约，结合依赖注入容器与多实现的基础设施层，实现了良好的关注点分离与扩展性。开发者应遵循“上层仅依赖接口、下层不依赖上层”的原则，利用容器进行依赖装配，借助仓储与事件总线实现业务编排与跨服务通信，从而在保证内聚的同时降低耦合度。
