引言​

本文件面向架构师与高级开发者，系统阐述 Sparrow 框架在代码层面如何落地“整洁架构”（Clean Architecture）。通过明确应用层、适配器层、领域层、基础设施层的职责边界与交互关系，解释依赖倒置原则、分层设计与模块化实现，给出架构决策的技术考量、权衡取舍与扩展性设计建议。文中提供多幅架构与类图，帮助读者快速把握系统全貌。

项目结构​

Sparrow 采用按“关注点”分层的目录组织方式，结合 Go 模块化特性，形成清晰的分层边界：

  • 领域层（entity）：承载业务实体与聚合根，定义业务不变量与行为契约
  • 用例层（usecase）：封装业务用例与流程编排，定义仓储接口与通用查询模型
  • 适配器层（adapter）：HTTP 路由与处理器、中间件、投影/Saga 等外部适配
  • 基础设施层（bootstrap、config、persistence、messaging、tasks 等）：提供运行时环境、配置、持久化、消息总线、任务调度等支撑能力

  • 应用程序入口与容器

    • App：全局应用对象，负责配置加载、日志初始化、HTTP 服务器启动、子进程生命周期管理、事件总线/消息总线接入、任务调度器集成
    • Container：轻量 IoC 容器，支持匿名与具名依赖注册与解析，实现依赖注入与单例缓存

  • 领域模型

    • BaseEntity：统一实体基类，提供标准字段与时间戳
    • AggregateRoot：聚合根接口，定义事件溯源、版本控制、业务校验等核心契约
    • Task：任务实体，体现领域层的业务对象

  • 用例与仓储

    • Repository[T]：用例层对持久化的抽象接口
    • BaseRepository[T]：仓储通用实现骨架，提供默认错误与查询选项工具
    • SessionService：会话管理用例，封装业务规则与仓储交互

  • 基础设施

    • Config：集中式配置结构，涵盖服务、日志、数据库、缓存、消息中间件等
    • SqlDBRepository[T]：基于 database/sql 的泛型仓储实现，支持软删除、事务、条件查询、分页、随机取样等
    • StreamHub：基于 NATS JetStream 的事件流总线，支持订阅、优雅关闭与启动
    • TaskScheduler：任务调度器接口，抽象任务的调度、取消、状态查询与生命周期管理

Sparrow 的整洁架构以“依赖倒置”为核心：上层（应用层、适配器层）仅依赖抽象（接口），下层（基础设施层）实现这些抽象；领域层独立于技术细节，仅暴露接口给用例层消费。下图展示关键组件与依赖方向：

详细组件分析​

依赖注入与容器（IoC）​

  • 设计要点
    • 支持匿名与具名提供者注册，解析时优先命中具名，否则回退匿名
    • 构造函数参数按类型解析依赖，自动缓存首次解析结果（单例）
    • 提供便捷的按名称解析与实例注入方法
  • 与应用层协作
    • App 通过 Container 注册与解析全局服务（如 NATS 连接、事件发布器、任务服务等）
    • 用例层通过 App 的解析方法获取仓储、会话服务等依赖

  • BaseEntity：统一实体标识与时间戳字段，为所有实体提供一致的生命周期管理

  • AggregateRoot：定义聚合根必须实现的接口，包括版本控制、事件溯源（命令处理、事件应用、未提交事件管理）、从事件流/快照重建状态、业务校验等

  • Task：任务实体，封装任务状态机与重试逻辑，体现领域层的业务不变量

  • 依赖方向

    • 上层（适配器、用例）仅依赖抽象（接口）
    • 下层（基础设施）实现抽象（具体仓储、消息总线、任务调度器）
    • 领域层不依赖任何技术细节，仅暴露接口给用例层

  • 耦合与内聚

    • 通过接口隔离与依赖注入，降低模块间耦合
    • 用例层与仓储接口分离，便于替换不同存储实现

  • 循环依赖

    • 通过接口与容器解耦，避免直接循环引用

  • 外部依赖

    • 通过 go.mod 明确第三方库范围，基础设施层围绕这些库进行适配

  • 仓储层

    • 事务与批量操作：SqlDBRepository 对批量保存/删除使用事务，减少往返开销
    • 条件查询与分页：支持多条件、排序、分页，避免一次性加载全量数据
    • 软删除：通过 deleted_at 字段实现软删除，兼顾审计与性能

  • 任务调度

    • 并发与顺序两种调度器，可根据任务特性选择执行模式
    • 支持最大并发限制，防止资源争用

  • 日志与配置

    • 集中式配置与可插拔日志，便于在生产环境调整性能相关参数

故障排查指南​

  • 启动与重试
    • App 提供子进程指数退避重试机制，最大重试次数与等待上限可控
    • 优雅关闭：统一的关闭流程与超时控制，确保资源释放
  • 仓储错误
    • BaseRepository 对空 ID、空字段等常见错误进行统一包装，便于定位问题
    • SqlDBRepository 对查询异常进行分类处理（如未找到、事务回滚等）
  • 会话服务
    • 过期检查与自动清理，避免脏数据影响业务
  • 消息总线
    • StreamHub 支持订阅器的启动/关闭，便于在调试阶段启用/禁用事件处理

Sparrow 通过整洁架构与依赖倒置，将业务逻辑与技术实现解耦，形成高内聚、低耦合的分层体系。用例层以接口抽象仓储，领域层专注于业务不变量，基础设施层提供可替换的实现，适配器层负责外部交互。配合 IoC 容器、集中式配置与任务/消息基础设施，系统具备良好的扩展性与可维护性。建议在扩展新功能时遵循“上层依赖抽象、下层实现抽象”的原则，持续保持架构清晰。
