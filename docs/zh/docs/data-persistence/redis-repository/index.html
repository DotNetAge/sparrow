简介​

本文件系统性阐述 Sparrow 中基于 Redis 的仓储实现，覆盖设计原理、数据结构选择、键空间设计、序列化策略、过期策略、集合/有序集合/哈希等 Redis 特有操作支持、管道执行优化、事务处理、连接配置、集群与哨兵模式配置、性能调优、内存监控与故障恢复机制。目标是为运维与开发团队提供一份可直接落地的 Redis 仓储使用指南。

项目结构​

围绕 Redis 仓储的关键模块分布如下：

  • 仓储实现：pkg/persistence/repo/redis.go
  • 事件总线（Redis Pub/Sub）：pkg/eventbus/redis/redis.go
  • Redis 配置模型：pkg/config/redis.go
  • 应用层数据库聚合：pkg/bootstrap/db.go
  • 应用启动与容器：pkg/bootstrap/app.go、pkg/bootstrap/container.go
  • 测试用例：pkg/persistence/repo/redis_test.go
  • RedisRepository[T]：泛型仓储实现，封装实体的 CRUD、批量操作、分页、条件查询、随机取样、TTL 管理等。
  • redisBus：基于 Redis Pub/Sub 的事件总线，负责事件发布与订阅。
  • RedisConfig：Redis 连接配置模型（主机、端口、密码、DB、事件存储专用 DB）。
  • Database：应用层聚合数据库客户端（Redis、SQL、Badger）。
  • Container：简单 IoC 容器，支持具名与匿名依赖解析。

Redis 仓储在应用启动阶段通过配置与容器注入 Redis 客户端，仓储实例持有客户端并以“键前缀:实体ID”的键空间组织数据；事件总线同样使用 Redis 客户端进行发布/订阅。仓储与事件总线均依赖统一的 Redis 配置模型。

RedisRepository[T] 设计与实现​

  • 泛型与接口契合：实现 usecase.Repository[T]，通过 BaseRepository[T] 扩展能力。
  • 键空间设计：键格式为“prefix:id”，prefix 由构造函数传入，确保不同实体类型隔离。
  • 序列化策略：统一采用 JSON 序列化/反序列化，便于跨语言与人类可读。
  • 过期策略：
    • 构造时可设置全局 TTL；若 >0，则 Set 操作携带 TTL。
    • 支持对特定键设置/查询 TTL（SetTTLForKey、GetTTLForKey），以及全局 TTL 查询与设置。
  • 数据结构选择：
    • 字符串存储：键值对存储完整实体 JSON。
    • 批量读写：MGET、DEL、EXISTS、KEYS 等命令组合实现高效批量操作。
  • 管道执行优化：SaveBatch 内部使用 Pipeline，减少 RTT，提升吞吐。
  • 条件查询与分页：FindWithConditions/FindByFieldWithPagination/CountWithConditions 基于内存过滤与排序，适用于中小规模数据集。
  • 随机取样：Random 通过 Keys + 随机打散 + MGET 实现近似随机取样。
  • 错误处理：统一包装 RepositoryError，明确实体类型、操作、ID 与错误信息。
  • 发布：将事件 JSON 序列化后发布到频道（事件类型作为主题）。
  • 订阅：为每个事件类型维护一个 PubSub 列表，启动协程异步处理消息。
  • 生命周期：Close 时关闭所有订阅与 Redis 连接，保证优雅关停。
  • 错误处理：订阅失败、消息解析失败、处理器异常均有日志记录与错误返回。

  • 键空间：prefix:id，prefix 由构造函数传入，确保不同实体类型隔离；内部自动补全末尾冒号。

  • 序列化：统一 JSON；反序列化失败返回错误，避免脏数据。

  • TTL：支持全局 TTL 与单键 TTL；GetTTLForKey 返回 -2 表示键不存在（Go-Redis 行为）。

  • 管道执行：SaveBatch 使用 Pipeline，一次性提交多条 Set 指令，显著降低网络往返。

  • 事务处理：当前仓储未显式使用 Multi/Exec；事件总线亦未使用事务。如需强一致性场景，可在上层业务封装 Multi/Exec。

仓储在 Redis 上实现“全表扫描 + 内存过滤/排序/分页”，适用于中小规模数据集。流程如下：

  • 当前实现：使用 redis.NewClient 与基本选项（地址、密码、DB）创建客户端。

  • 集群/哨兵：代码中未显式使用 go-redis 的 ClusterOptions/SentinelOptions。若需集群/哨兵，请在应用初始化处替换客户端创建逻辑，传入相应 Options。

  • Database 聚合 RedisClient，供仓储与事件总线使用。

  • App 启动时创建 Database 并注入容器，仓储通过容器解析获得 RedisClient。

  • 容器支持具名与匿名依赖解析，便于多仓库/多客户端场景。

  • 仓储依赖 Redis 客户端与配置模型，提供统一的实体操作接口。

  • 事件总线依赖 Redis 客户端与配置模型，提供发布/订阅能力。

  • 应用层通过容器注入 Database，仓储与事件总线共享同一 RedisClient。

  • 管道优化：SaveBatch 使用 Pipeline，减少 RTT，适合批量写入。

  • 批量读取：FindAll/FindByIDs 使用 MGET，减少多次网络往返。

  • 键空间隔离：通过前缀隔离不同实体类型，避免键冲突。

  • TTL 策略：合理设置全局 TTL 与单键 TTL，结合业务生命周期控制内存占用。

  • 条件查询：当前为内存过滤，建议在大规模数据场景下引入索引或外部搜索引擎。

  • 连接复用：仓储与事件总线共享同一 RedisClient，避免重复连接。

  • 连接失败：检查 RedisConfig 的 Host/Port/Password/DB；事件总线在创建时会 Ping 校验连接。

  • 键不存在：FindByID 返回 RepositoryError，message 指明实体未找到；GetTTLForKey 返回 -2 表示键不存在。

  • 批量操作异常：SaveBatch/SaveBatch/DeleteBatch 会在任一实体失败时返回错误；检查实体 ID 与 JSON 序列化。

  • 条件查询无结果：确认字段名与 JSON 标签一致；Operator 支持 EQ/NEQ/GT/GTE/LT/LTE/LIKE/IN/IS_NULL/IS_NOT_NULL。

  • 随机取样：当实体数量小于取样数时，返回全部实体；注意 Keys/随机打散的开销。

Sparrow 的 Redis 仓储以简洁的键空间设计与 JSON 序列化为核心，结合管道优化与批量读写，满足常见实体持久化需求。事件总线基于 Redis Pub/Sub 提供轻量级解耦通信。对于大规模数据与强一致性场景，建议引入索引/搜索引擎与事务封装；对于高可用与弹性扩展，建议在应用层替换为集群/哨兵客户端。

附录​

Redis 特有操作支持现状​

  • 集合/有序集合/哈希：当前仓储未直接使用这些数据结构；如需使用，可在上层业务封装相应命令。

  • 管道执行：已使用 Pipeline 优化批量写入。

  • 事务处理：未显式使用 Multi/Exec；如需强一致，可在业务层封装。

  • 当前配置模型仅包含基础字段；如需集群/哨兵，应在应用初始化处创建相应 Options 并注入仓储与事件总线。

  • 建议在生产环境启用连接池参数（如最小空闲连接、最大空闲连接、连接超时等）以提升稳定性。

  • 使用 testcontainers 启动 Redis 容器，确保测试环境一致性。

  • 覆盖 Save/Find/Delete/批量操作/分页/条件查询/TTL/随机取样等关键路径。

  • 验证错误场景（空 ID、不存在键、反序列化失败等）。
