简介​

本文件系统性阐述 Sparrow 框架中的实体设计，重点覆盖：

  • BaseEntity 基类的设计理念与职责
  • Entity 接口的约束与契约
  • 实体标识符生成机制
  • 实体生命周期管理、状态变更方法与时间戳处理
  • Task 实体的完整实现示例与最佳实践
  • 验证规则、数据完整性保障与性能优化建议
  • 事件溯源与聚合根的扩展能力

项目结构​

实体相关代码集中在 pkg/entity 目录，采用“接口 + 基类 + 具体实体”的分层设计，配合命令、领域事件与聚合根接口，形成清晰的 DDD/CQRS 与事件溯源架构。

  • Entity 接口：统一约束实体的标识、时间戳访问与更新能力，确保仓储与持久化层的契约一致。
  • BaseEntity 基类：提供 Id、CreatedAt、UpdatedAt 的标准字段与访问器，统一实体的生命周期管理。
  • Task 实体：以任务为核心业务实体，展示状态机、重试策略、执行时长计算与过期判断等业务逻辑。
  • 聚合根接口与基类：提供版本控制、事件收集与应用、快照加载等事件溯源能力。
  • 领域事件与基础事件：标准化事件元数据与负载处理。
  • 命令与基础命令：标准化命令的标识与聚合根绑定。

实体设计遵循整洁架构的“核心层”理念，将业务实体置于最内层，通过接口与基类向上提供稳定契约，向下由具体实体实现业务规则。聚合根在事件溯源模式下承担状态变更与事件管理职责；命令驱动聚合根产生领域事件；仓储负责实体的持久化；事件存储与快照用于状态重建与优化。

BaseEntity 基类与 Entity 接口​

  • 设计理念
    • 统一实体标识与时间戳管理，避免重复实现。
    • 通过 NewBaseEntity 初始化创建与更新时间，确保一致性。
  • 关键方法
    • Get/SetID：读写实体标识
    • Get/SetCreatedAt：只读创建时间
    • Get/SetUpdatedAt：更新时间随业务变更同步
  • 生命周期
    • 创建时设置 CreatedAt 与 UpdatedAt
    • 任何状态变更应调用 SetUpdatedAt 以保持审计一致性
  • 状态枚举：pending、running、completed、failed、cancelled
  • 关键方法
    • Start/Complete/Fail/Cancel：驱动状态机与时间戳更新
    • CanRetry：基于 Retries 与 MaxRetry 的重试判定
    • GetExecutionTime：计算执行时长（需两端时间均存在）
    • IsExpired：基于 CreatedAt 的过期检查
  • 标识符生成
    • NewTask 内部调用 generateTaskID，组合纳秒与毫秒，保证高并发下的唯一性
  • 时间戳处理
    • 所有状态变更均更新 UpdatedAt，Fail 时增加 Retries
  • AggregateRoot 接口
    • 继承 Entity，扩展聚合类型、聚合 ID、版本控制与事件溯源能力
    • 提供 HandleCommand、ApplyEvent、LoadFromEvents、LoadFromSnapshot、Validate 等预留方法
  • BaseAggregateRoot 基类
    • 嵌入 BaseEntity，维护 Version 与 uncommittedEvents 列表
    • AddEvent：校验事件 ID/类型/聚合 ID 一致性，递增版本并更新 UpdatedAt
    • MarkEventsAsCommitted：清空未提交事件列表
    • HasUncommittedEvents：辅助判断
  • 事件与快照
    • DomainEvent 接口定义事件元数据
    • BaseEvent 提供事件的通用结构与负载解码工具

  • Command 接口与 BaseCommand：标准化命令 ID 与聚合根 ID 的绑定

  • BaseEvent：标准化事件元数据，提供负载解码工具 UnmarshalEvent

  • Session：基于 BaseEntity，提供过期检查、数据读写与 TTL 更新

  • Transaction：实现 Entity 接口，用于 Saga 协作的事务编排

  • 组件耦合

    • 实体层依赖 Entity 接口与 BaseEntity 基类，确保一致性
    • 聚合根层在事件溯源模式下依赖 DomainEvent 接口与 BaseEvent 基类
    • 命令层与聚合根交互，通过事件实现无侵入的状态变更

  • 外部依赖

    • UUID 工具用于生成全局唯一标识
    • JSON 编解码用于事件负载传输
  • 标识符生成
    • Task 使用纳秒+毫秒组合生成 ID，具备高并发唯一性；如需进一步降低冲突概率，可在上层引入分布式 ID 生成器（例如雪花算法）或数据库自增主键策略
  • 时间戳更新
    • 所有状态变更均调用 SetUpdatedAt，确保审计一致性；批量更新时可合并一次更新，减少多次写入
  • 事件溯源
    • 使用 BaseAggregateRoot 的 AddEvent 自动递增版本并更新时间戳；建议对未提交事件进行批量化提交，减少事件存储压力
  • 查询与排序
    • 实体实现 GetCreatedAt 可用于排序与分页；建议在仓储层对常用查询建立索引，避免全表扫描
  • 内存与序列化
    • 领域事件的 Payload 建议采用二进制或压缩编码，降低网络传输与存储开销

故障排查指南​

  • 事件 ID/类型/聚合 ID 校验失败
    • AddEvent 会对事件 ID、类型与聚合 ID 进行严格校验，若出现 panic，请检查事件构造与聚合根 ID 是否一致
  • 重复事件 ID
    • AddEvent 会检测重复事件 ID，避免事件流污染；请确保事件 ID 唯一
  • 状态变更异常
    • Task 的 Fail 会增加 Retries，若达到 MaxRetry 则不再自动重试；请检查 CanRetry 条件与业务流程
  • 过期与执行时长
    • IsExpired 基于 CreatedAt 判断；GetExecutionTime 需两端时间均存在，否则返回空

Sparrow 框架的实体设计以接口与基类为核心，通过统一的时间戳管理、严格的事件溯源与聚合根版本控制，提供了高内聚、低耦合的领域建模能力。Task 实体展示了状态机与重试策略的典型实现，Session 与 Transaction 则体现了不同层次的业务抽象。结合模板化聚合根生成与事件存储/快照机制，开发者可快速构建可演进、可观测的领域模型。

附录​

    • 通过聚合根的 ApplyEvent 与事件存储实现业务可追溯
