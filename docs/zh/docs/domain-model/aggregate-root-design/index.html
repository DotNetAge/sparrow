简介​

本文件围绕领域驱动设计中的“聚合根”展开，结合代码库中 Session 聚合根的实现，系统阐述聚合根的边界划分原则、与实体的关系、一致性保证机制、事务边界管理、业务规则封装、生命周期与状态同步、并发控制、职责分离、依赖注入与测试策略。目标读者为架构师与高级开发者。

项目结构​

本项目采用整洁架构分层：接口适配器层（HTTP）、用例层（业务服务）、实体层（实体与聚合根）、基础设施层（仓储、事件存储、消息总线）。Session 聚合根位于实体层，SessionService 位于用例层，HTTP 处理器位于接口适配器层，仓储与事件存储位于基础设施层。

  • 聚合根接口与基础实现
    • 聚合根接口定义了聚合标识、版本控制、事件溯源支持、状态加载与业务验证等能力。
    • 基础聚合根提供版本号、未提交事件列表、事件追加与标记等通用逻辑。
  • 实体与基础实体
    • 实体接口统一 ID、创建/更新时间；基础实体提供默认字段与方法。
  • 会话实体
    • 会话实体嵌入基础实体，包含数据字典与过期时间，提供过期判断与数据读写方法。
  • 用例服务
    • 会话服务封装业务流程：创建、获取、保存、删除、更新数据、清理过期会话。
  • 仓储与事件存储
    • 基础仓储定义通用 CRUD 与查询接口；内存仓储提供并发安全的实现。
    • 事件存储接口定义事件持久化、查询、快照与批量操作能力。
  • HTTP 处理器
    • 提供会话的 HTTP API：创建、获取、更新数据、删除、获取特定数据。
  • 依赖注入容器与应用启动
    • 容器支持匿名/具名注册与解析；应用负责启动 HTTP 服务、子进程与资源清理。

Session 聚合根遵循 DDD 聚合边界：以会话为核心，通过服务层协调仓储与事件存储，保障业务规则与一致性；HTTP 层负责请求接入与响应输出。

聚合根接口与基础实现​

  • 聚合根接口
    • 聚合标识：聚合类型与聚合 ID。
    • 版本控制：版本号获取、设置、递增，用于乐观并发控制。
    • 事件溯源：处理命令生成事件、应用事件、获取未提交事件、标记已提交、从事件流/快照加载状态。
    • 业务验证：预留 Validate 接口。
  • 基础聚合根
    • 维护 BaseEntity、版本号与未提交事件列表。
    • 提供 AddEvent/AddUncommittedEvents、MarkEventsAsCommitted、HasUncommittedEvents 等方法。
    • 对事件 ID、类型、聚合 ID 与重复事件进行严格校验，防止事件流破坏。
  • 会话实体
    • 嵌入基础实体，包含数据字典与过期时间。
    • 提供过期判断、更新过期时间、设置/获取/删除数据等方法。
  • 会话服务
    • 业务流程：创建会话、获取会话并检查过期、保存会话（自动更新过期时间）、删除会话、更新数据、获取数据、清理过期会话。
    • 通过仓储实现持久化，日志记录错误。
  • 事务边界
    • 服务层以单个业务操作为事务边界，确保同一业务语义下的原子性与一致性。
    • 过期检查与删除在获取会话时完成，避免脏读。
  • 事件模型
    • 领域事件接口包含事件 ID、类型、聚合 ID/类型、版本与时间戳。
    • 基础事件结构体提供默认实现与序列化辅助。
  • 事件存储接口
    • 基础能力：保存事件、读取事件、加载聚合状态、关闭。
    • 查询能力：按版本、类型、时间范围查询与分页。
    • 快照能力：保存/获取最新快照。
    • 批量能力：批量保存事件。
  • 一致性与并发控制
    • 聚合根版本号用于乐观锁；事件追加时递增版本并设置事件版本。
    • 事件 ID、类型与聚合 ID 校验，防止重复与错配。
    • 未提交事件列表用于事件存储与发布，提交后清空。
  • 基础仓储
    • 定义泛型实体的通用接口，提供保存、查找、更新、删除、分页、统计与条件查询等方法。
    • 多处对空 ID 进行校验，避免非法操作。
  • 内存仓储
    • 基于读写锁实现并发安全的 CRUD 与批量操作。
    • 自动设置创建/更新时间，支持条件查询与随机选取。
    • 提供 Exists、Clear 等辅助方法，便于测试与运维。
  • 容器
    • 支持匿名与具名依赖注册与解析，构造函数参数自动解析，实例缓存（单例）。
  • 应用启动
    • 初始化配置与日志，创建 Gin 引擎，注册容器，启动子进程与 HTTP 服务器。
    • 提供 GetSessions 等便捷解析方法，支持 NATS 连接与事件总线/发布器。

  • 单元测试

    • 使用内存仓储替代真实数据库，利用并发安全与条件查询能力快速验证业务逻辑。
    • 对会话服务的关键路径（创建、获取、保存、删除、更新数据、清理过期）进行断言。

  • 集成测试

    • 通过容器注册与解析，组装端到端流程，验证 HTTP -> 服务 -> 仓储 -> 事件存储链路。

  • 行为测试

    • 使用模拟聚合根与事件总线，验证事件发布、重放与快照恢复的正确性。

  • 组件耦合

    • 会话服务依赖仓储接口，解耦具体实现；HTTP 处理器依赖服务接口，便于替换与测试。
    • 聚合根接口与基础聚合根提供统一的事件溯源与版本控制能力，降低各聚合根实现成本。

  • 外部依赖

    • 事件存储接口支持数据库型与事件流两种实现，事件流模式下侧重读能力与发布。
    • NATS 连接与事件总线用于事件发布与订阅，支撑最终一致性。
  • 事件存储
    • 数据库型事件存储适合强一致场景，需关注并发写入与批量操作性能。
    • 事件流型存储将并发控制交给平台，适合高吞吐与最终一致场景。
  • 仓储
    • 内存仓储适合测试与低延迟场景；生产环境建议使用关系型/文档型数据库或键值存储。
    • 条件查询与分页在内存仓储中通过反射实现，复杂查询建议迁移至专用数据库。
  • 并发控制
    • 基于版本号的乐观锁减少锁竞争；事件追加时严格校验避免重复事件。
  • 事件重放与快照
    • 对历史事件较多的聚合根，使用快照可显著提升重建性能。

故障排查指南​

  • 常见错误
    • 会话不存在：获取会话失败或过期删除后返回不存在。
    • 非法会话：保存空会话返回无效会话错误。
    • 仓储错误：空 ID、实体不存在、已存在等错误信息。
  • 排查步骤
    • 检查 HTTP 请求参数与路径变量是否正确。
    • 查看服务日志定位错误来源（创建、获取、保存、删除）。
    • 验证仓储实现（内存/数据库）是否正确设置时间戳与 ID。
    • 若使用事件存储，确认事件 ID/类型与聚合 ID 一致性，检查版本号与未提交事件列表。

本设计以 Session 聚合根为例，展示了 DDD 聚合根在整洁架构中的职责与实现要点：清晰的聚合边界、严格的业务规则封装、基于版本号的乐观并发控制、事件溯源与最终一致性的事件存储方案，以及通过依赖注入与容器实现的可替换与可测试架构。该模式可推广至其他聚合根，建议在实际项目中结合事件流与数据库型事件存储，平衡一致性与性能。
