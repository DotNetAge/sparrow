简介​

本文件面向任务系统的设计与实现，聚焦于以下目标：

  • 明确 Task 接口的设计与实现要求，包括 ID()、Type()、Schedule()、Handler()、OnComplete()、OnCancel()、IsRecurring()、GetInterval()、GetTimeout() 等方法的职责与规范。
  • 解释 TaskSchedule 结构体的三种调度类型：即时执行、定时执行、周期性执行的配置方法与使用场景。
  • 对比 TaskScheduleType 枚举与 ExecutionMode 枚举的区别与应用场景。
  • 总结任务配置的最佳实践与常见使用模式。

项目结构​

任务系统位于 pkg/tasks 目录，围绕“任务接口 + 构建器 + 调度器 + 包装器”的分层设计组织，配合 pkg/entity 中的任务实体模型，形成清晰的职责边界与可扩展性。

  • Task 接口：统一抽象任务行为，定义任务标识、类型、调度时间、处理器、完成/取消回调、周期性属性与超时等。
  • TaskSchedule：封装调度类型与时间/间隔信息，提供 ImmediateExecution、ScheduleAt、ScheduleRecurring 便捷构造。
  • TaskScheduleType：调度类型枚举，区分即时、定时、周期性。
  • ExecutionMode：执行模式枚举，区分并发与顺序，用于混合调度器按任务类型选择执行路径。
  • TaskBuilder：链式配置任务，负责校验与组装 Task 实例。
  • SchedulerWrapper：对外统一入口，封装调度器能力，提供 RunTask*、RunTypedTask*、CancelTask、GetTaskStatus、ListTasks 等。
  • 调度器实现：ConcurrentScheduler（并发）、SequentialScheduler（顺序）、HybridScheduler（按类型选择）。

任务从“构建器”生成“任务对象”，经“调度器包装器”提交至“调度器”。调度器根据任务类型与调度配置选择执行模式，并在执行过程中处理超时、重试、完成回调与取消回调。

Task 接口设计与实现规范​

  • 方法职责

    • ID(): 返回任务唯一标识，用于调度器索引与取消。
    • Type(): 返回任务类型，用于混合调度器按类型选择执行模式。
    • Schedule(): 返回任务计划执行时间；即时任务通常为当前时间或零值。
    • Handler(): 返回任务处理函数，接收 context.Context，返回 error。
    • OnComplete(): 返回任务完成后回调，接收 context.Context 与 error。
    • OnCancel(): 返回任务取消后回调，接收 context.Context。
    • IsRecurring(): 判断是否为周期性任务。
    • GetInterval(): 返回周期性任务间隔。
    • GetTimeout(): 返回任务超时时间，用于执行上下文超时控制。

  • 实现要点

    • builtTask 同时实现 Task 与 TaskInfoProvider，便于调度器读取 TaskInfo。
    • Schedule() 的计算在 Build() 时确定，即时任务为当前时间，定时任务为指定时间，周期性任务为当前时间加上间隔。
    • IsRecurring()/GetInterval() 由 TaskSchedule 决定，周期性任务在顺序调度器中会重新计算下次执行时间并回写。

  • 结构体字段

    • Type: TaskScheduleType（即时/定时/周期性）
    • At: 定时执行的时间点
    • Interval: 周期性执行的间隔

  • 便捷构造

    • ImmediateExecution(): 即时执行
    • ScheduleAt(at): 指定时间执行
    • ScheduleRecurring(interval): 周期性执行

  • 使用场景

    • 即时执行：快速响应的轻量任务
    • 定时执行：延时任务、定时批处理
    • 周期性执行：心跳、轮询、周期性清理

  • TaskScheduleType

    • 描述“何时执行”：即时、定时、周期性
    • 由 TaskSchedule 决定，影响调度器如何入队与重排

  • ExecutionMode

    • 描述“如何执行”：并发、顺序
    • 由混合调度器按任务类型映射决定，影响调度器选择并发或顺序执行器

  • 应用建议

    • 定时/周期性任务默认顺序执行，保证时序与一致性
    • 高吞吐、无强时序依赖的任务使用并发执行
    • 通过 SchedulerWrapper 的 WithSequential/WithConcurrent 对任务类型进行粒度化控制

  • 链式配置

    • WithID/WithType/WithHandler/WithOnComplete/WithOnCancel
    • WithSchedule/WithRecurring/WithTimeout/WithRetry/WithPriority

  • 构建与校验

    • 必填项：Handler 不可为空
    • 定时任务：执行时间必须晚于当前时间
    • 重试与超时：均不可为负数
    • 计算执行时间：即时为当前时间，定时为 At，周期性为当前时间+Interval

  • 输出：实现 Task 与 TaskInfoProvider 的 builtTask

  • 功能

    • RunTask*/RunTypedTask*：提交任务（即时/定时/周期性）
    • CancelTask：取消任务
    • GetTaskStatus/ListTasks：查询状态与列表
    • SetTaskTypeMode：按任务类型设置执行模式
    • SetMaxConcurrentTasks：设置并发调度器最大并发

  • 默认行为

    • 默认重试次数：3 次（可通过 WithRetryCount 调整）
    • 默认调度器：混合调度器（按类型选择并发/顺序）

  • 最佳实践

    • 明确任务类型，使用 RunTypedTask* 并结合 WithSequential/WithConcurrent 精准控制
    • 对高吞吐任务设置合理的并发上限
    • 对需要严格时序的任务（如定时/周期性）使用顺序执行
    • 为关键任务设置超时，避免阻塞

  • ConcurrentScheduler

    • 并发执行，受 maxConcurrent 限制
    • 支持超时、重试、TTL 清理、死信状态
    • 即时任务立即执行，定时/周期性任务在监视器中触发

  • SequentialScheduler

    • 顺序执行，维护待执行队列与完成通知通道
    • 定时任务到达时间后加入队列，周期性任务完成后重新计算下次执行时间
    • 指数退避重试（顺序调度器）

  • HybridScheduler

    • 按任务类型映射选择并发或顺序
    • 定时/周期性任务默认顺序
    • 支持动态设置任务类型执行模式

  • Task 实体（领域层）

    • 字段：ID、Type、Status、Payload、Result、Error、Priority、Retries、MaxRetry、StartedAt、EndedAt
    • 方法：Start/Complete/Fail/Cancel/CanRetry/GetExecutionTime/IsExpired

  • 状态机（调度器侧）

    • unknown/waiting/running/completed/cancelled/failed/retrying/dead_letter
    • 重试策略：并发调度器线性退避，顺序调度器指数退避
    • 死信：超过最大重试次数或不可重试时进入 dead_letter

  • 并发与顺序的选择

    • 并发：提升吞吐，适用于无强依赖与无时序要求的任务
    • 顺序：保证时序与一致性，适用于定时/周期性任务

  • 并发上限

    • 通过 SetMaxConcurrentTasks 控制并发度，避免资源争用

  • 重试策略

    • 并发调度器采用线性退避，顺序调度器采用指数退避，避免雪崩

  • 内存管理

    • TTL 与最大任务数限制，定期清理已完成/取消/失败任务，防止内存膨胀

  • 超时控制

    • 通过 Task.GetTimeout() 为任务执行设置上下文超时，避免长时间阻塞

  • 常见错误与定位

    • “任务处理器不能为空”：检查 TaskBuilder.WithHandler 是否设置
    • “定时任务的执行时间必须大于当前时间”：确认 ScheduleAt 的时间参数
    • “任务ID不能为空”：并发调度器对任务ID有约束
    • “任务不存在”：确认任务ID正确或已被清理

  • 状态异常

    • failed/retrying/dead_letter：查看 LastError 与 NextRetryAt，调整重试策略
    • cancelled：确认是否被主动取消或超时取消

  • 调试建议

    • 使用 ListTasks 获取全量任务快照
    • 使用 GetTaskStatus 查询特定任务状态
    • 为关键任务设置超时，避免长时间阻塞
    • 对顺序任务关注队列长度与执行耗时，避免积压

本任务系统通过清晰的接口抽象、灵活的构建器与多调度器实现，提供了从“即时/定时/周期性”到“并发/顺序”的完整任务生命周期管理。配合 SchedulerWrapper 的统一入口与默认重试策略，能够满足大多数业务场景的需求。建议在生产环境中明确任务类型与执行模式，合理设置并发上限与重试策略，并通过 TTL 与状态监控保障系统稳定性。

附录​

任务配置最佳实践清单​

  • 明确任务类型：使用 RunTypedTask* 并结合 WithSequential/WithConcurrent
  • 合理设置超时：为耗时任务设置 GetTimeout，避免阻塞
  • 控制并发：通过 SetMaxConcurrentTasks 限制并发度
  • 重试策略：根据业务容忍度设置 WithRetryCount，注意退避策略差异
  • 定时/周期性：优先使用顺序执行，确保时序与一致性
  • 监控与清理：定期检查 ListTasks，关注 dead_letter 与 retrying 状态

常见使用模式​

  • 即时任务：RunTask/RunTaskWithID
  • 延迟任务：RunTaskAfter/RunTaskAfterWithID
  • 定时任务：RunTaskAt/RunTaskWithIDAt
  • 周期性任务：RunTaskRecurring/RunTaskWithIDRecurring
  • 类型化任务：RunTypedTask/RunTypedTaskWithID
