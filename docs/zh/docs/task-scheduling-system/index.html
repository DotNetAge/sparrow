简介​

本文件为 Sparrow 任务调度系统的详细技术文档，覆盖并发调度器、顺序调度器、混合调度器的设计原理与实现细节；阐述任务生命周期管理、状态跟踪、优先级处理与重试机制；提供任务构建器与调度器包装器的使用方法；包含监控与统计信息、最佳实践、性能优化与故障恢复策略，并展示如何扩展调度器以支持自定义调度算法与任务类型。文档面向任务调度开发者，提供从概念到实现的完整指导。

项目结构​

任务调度系统位于 pkg/tasks 目录，采用分层设计：

  • 接口与模型：定义任务调度器接口、任务模型、状态与调度类型
  • 调度器实现：并发、顺序、混合三种调度器
  • 构建器与包装器：简化任务构建与统一调度入口
  • 测试：覆盖并发、顺序、混合、重试、超时、内存管理等场景
  • 任务调度器接口：统一的调度、取消、状态查询、并发控制与生命周期管理能力
  • 任务模型：任务信息、状态、调度类型、执行模式、重试配置
  • 任务构建器：链式配置任务的 ID、类型、处理器、回调、调度时间、超时、重试与优先级
  • 调度器包装器：统一入口，负责任务提交、模式选择、并发控制与默认重试
  • 三种调度器实现：并发、顺序、混合，分别满足不同执行需求与策略

调度系统采用“接口抽象 + 多实现 + 包装器入口”的架构：

  • 接口层定义统一能力
  • 实现层提供并发/顺序/混合三种策略
  • 包装器层提供易用 API 与默认行为（如默认重试、并发数设置、任务类型模式映射）

并发调度器（concurrentScheduler）​

并发调度器允许多任务并行执行，支持最大并发数限制、任务 TTL、内存清理与重试机制。其关键特性包括：

  • 并发槽位控制：通过活跃任务计数与条件变量/轮询实现并发节流
  • 任务监视：定时扫描等待与重试中的任务，触发执行
  • 重试策略：线性递增延迟（上限 500ms），达到最大重试次数后进入死信
  • 内存管理：达到最大任务数时移除最老的已完成/取消/死信任务；TTL 过期后清理
  • 任务取消：调用取消回调并标记状态为取消

顺序调度器保证同一时刻仅有一个任务执行，适用于定时与周期性任务。其关键特性包括：

  • 有序执行：维护待执行队列，按序取出并执行
  • 任务队列与通知：使用带缓冲队列与完成通知通道协调执行
  • 监视器：定时检查定时任务是否到达执行时间并加入队列
  • 重试策略：指数退避（上限 1 分钟），达到最大重试次数后标记失败
  • 内存清理：按更新时间移除最老的已完成/取消/失败任务

混合调度器根据任务类型与调度时间选择执行模式：

  • 任务类型映射：可将特定任务类型绑定为顺序或并发模式
  • 调度决策：若任务类型为顺序或为定时/周期任务则走顺序调度器，否则走并发调度器
  • 统一接口：封装并发与顺序调度器，提供统一的调度、取消、状态查询与列表

任务构建器提供链式配置，支持：

  • ID、类型、处理器、完成/取消回调
  • 调度时间（即时、定时、周期）、超时、重试次数、优先级
  • 构建后生成内置任务对象，实现 Task 与 TaskInfoProvider 接口

调度器包装器提供统一入口与默认行为：

  • 任务提交：支持立即、定时、周期、延迟执行与带类型任务
  • 模式映射：可配置任务类型到执行模式的映射
  • 并发控制：支持设置最大并发数（对并发/混合调度器生效）
  • 默认重试：通过选项注入默认重试次数
  • 生命周期：统一 Start/Stop/Close
  • 接口耦合：所有调度器实现 TaskScheduler 接口，便于替换与组合
  • 组合关系：混合调度器组合并发与顺序调度器；包装器持有调度器实例
  • 任务内聚：builtTask 内嵌 TaskInfo，实现 Task 与 TaskInfoProvider，降低跨层耦合
  • 外部依赖：测试中使用 testify 断言与原子计数器，无外部运行时依赖

  • 并发控制

    • 并发调度器使用活跃任务计数与条件变量/轮询控制并发槽位，避免过度竞争
    • 混合调度器可针对特定任务类型设置并发模式，平衡吞吐与一致性

  • 任务 TTL 与内存管理

    • 并发/顺序调度器均提供 TTL 与最大任务数限制，防止内存膨胀
    • 定时清理过期任务，减少无效数据占用

  • 重试策略

    • 并发调度器采用线性递增延迟（上限 500ms），顺序调度器采用指数退避（上限 1 分钟）
    • 达到最大重试次数后进入死信/失败状态，避免无限重试

  • 超时控制

    • 任务执行支持超时上下文，超时或取消时标记为取消而非失败

  • 顺序执行的代价

    • 顺序调度器串行化执行，适合对时序敏感的任务，但吞吐较低

  • 任务状态异常

    • 并发/顺序调度器均提供 GetTaskStatus 与 ListTasks，可用于诊断
    • 若任务长时间处于重试中，检查重试延迟与最大重试次数配置

  • 并发阻塞

    • 并发调度器通过活跃任务计数与等待机制控制并发，若出现阻塞，检查最大并发数设置与任务执行时间

  • 任务丢失

    • 检查任务 TTL 与最大任务数限制，确认是否被清理

  • 取消无效

    • 确认任务 ID 正确且任务仍处于等待状态；顺序调度器会从待执行队列移除任务

Sparrow 任务调度系统通过清晰的接口抽象与多实现策略，提供了灵活的并发、顺序与混合执行模式。配合任务构建器与调度器包装器，开发者可以快速构建、配置与管理任务。系统具备完善的重试、超时、内存管理与状态跟踪能力，适合在高并发与对时序敏感的场景中使用。通过扩展接口与模式映射，可进一步支持自定义调度算法与任务类型。

附录​

任务生命周期与状态流转​

  • 并发调度器状态流转：等待 -> 运行 -> 完成/重试/死信

  • 顺序调度器状态流转：等待 -> 运行 -> 完成/重试/失败

  • 死信与失败的区别：死信通常表示达到最大重试次数或不可恢复错误

  • 任务优先级：构建器支持优先级设置（1-10，1 最高），可在构建阶段配置

  • 调度类型：即时、定时、周期，分别对应不同的执行策略

  • 状态查询：通过包装器或调度器的 GetTaskStatus/ListTasks 获取任务状态与列表

  • 内存统计：通过 ListTasks 的长度与 TTL 清理频率评估内存占用

  • 性能指标：可通过并发调度器的最大并发数与活跃任务计数估算吞吐

  • 新增调度器：实现 TaskScheduler 接口，即可接入包装器或混合调度器

  • 自定义模式映射：通过包装器的模式映射选项，将任务类型绑定到新调度器

  • 自定义重试策略：在任务构建阶段配置重试次数与超时，或在任务处理器中实现自定义重试逻辑

  • 任务类型划分：将对时序敏感的任务（如支付、订单）归类为顺序执行

  • 并发数设置：根据硬件资源与任务特性设置最大并发数，避免资源争用

  • 超时与重试：为长耗时任务设置合理超时与重试策略，避免阻塞

  • 内存管理：合理设置 TTL 与最大任务数，定期清理过期任务

  • 监控与告警：通过状态查询与日志记录，建立任务执行监控

  • 应用启动时初始化调度器包装器，注入默认并发数与任务类型映射

  • 通过包装器统一提交任务，支持定时、周期与延迟执行

  • 优雅关闭：在应用关闭流程中调用包装器的 Close 方法
